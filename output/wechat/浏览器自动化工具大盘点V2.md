# 浏览器自动化工具盘点

现在越来越多 AI Agent 需要操作浏览器，但工具选哪个一直是个让人头疼的事。我最近把市面上用得比较多的几个方案都试了一遍，说说各自的脾气和适用场景。

## 先说结论

- 日常浏览、快速操作、省钱优先 → 选 **Agent Browser**
- 跑全流程测试、要求稳定 → 选 **Playwright MCP**
- 调试排错、性能分析、抓请求 → 选 **Chrome DevTools MCP**
- 深层页面钻取、多层点击 → 选 **Browser Use**
- 绕过 GUI 直接调 API → 选 **OpenCLI**

## 1. Agent Browser

来自 Vercel Labs，GitHub：https://github.com/vercel-labs/agent-browser。核心做法是用 Snapshot + Refs 机制，只给 AI 传必要的信息，不把整个 DOM 扔过去。

省 Token 这块确实厉害。官方数据是减少 93% 的 Token 消耗，我自己用下来体感也差不多。对 200k 上下文窗口的模型来说，省下来的空间可以多干不少活。

操作方式也简单。它会给页面元素编号（`@e1`、`@e2`），AI 拿到编号就能操作，不用折腾 CSS 选择器。速度也快，底层是轻量 CLI，没那么多多余交互。

但能力边界也明显：只支持 Chromium，遇到深层性能调试或者代码级排错就不够用了。适合"帮我打开看看"、"截个图"、"填个表"这类轻量操作，不适合干重活。

## 2. Playwright MCP

微软出品，GitHub：https://github.com/microsoft/playwright-mcp。基于 Accessibility Tree，定位是通用浏览器自动化加 AI 扩展。

它的强项在于专业和完整。Chrome、Firefox、WebKit 都支持，代码生成、测试录制、网络请求控制、设备模拟（143+ 预设）这些测试功能一应俱全。跑复杂的全流程自动化，稳定性没得说。

问题在于太重了。自带 32 个工具，光定义就吃掉 13.6k Token。加上运行时暴露的可访问性树，上下文窗口很容易被撑满，AI 干着干着就"记不住"前面的内容了。而且每步都要"看一眼、想一下、动一下"，长流程跑起来效率偏低。

所以这个工具更适合正式的测试场景：回归测试、E2E、用户旅程验证。能容忍慢，但要求稳，选它没错。

## 3. Chrome DevTools MCP

Google 出的，GitHub：https://github.com/ChromeDevTools/chrome-devtools-mcp，直接封装 Chrome DevTools Protocol。

这个工具的特点是"贴近底层"。因为是直接基于 CDP，它能做很多别的工具做不到的事：监听网络请求的头部、body、状态码、时序，抓 Console 报错栈，甚至逆向分析第三方页面的追踪脚本。做性能分析的时候，它能记录追踪数据、提取 Core Web Vitals 指标，这是别的方案基本没有的能力。

另一个跟 Playwright 不一样的地方是，它不是一次性跑脚本的模式，而是保持持续连接，可以反复操作。调试的时候这种模式更顺手。

缺点是只支持 Chrome，Token 消耗中等，工具定义也不少，但比 Playwright 好一些。日常开发中涉及调试、性能分析、网络抓包的，这个工具基本绕不开。

## 4. Browser Use

社区开源项目，GitHub：https://github.com/browser-use/browser-use。底层封装了 Playwright/Puppeteer，加上 AI 推理，目标是让 AI 像人一样"深度浏览"网页。

它的强项是"钻"得深。比如你要从列表页逐个点进详情、打开 PDF、追到 GitHub 读完再返回继续下一篇——这种层层点击、深入挖掘的场景，其他工具搞起来很费劲，Browser Use 能处理。AI 可以根据页面实际情况自己判断下一步点哪里，自主性比较强。

代价也不小。因为它严重依赖 GUI 截图识别，界面稍微改版就可能翻车。每一步都要调多模态模型推理，Token 烧得快，响应也慢。用之前要心里有数：它确实能帮你钻到最深处，但过程不会快，也不会便宜。

## 5. OpenCLI

开发者是 Jackwener（Apache Arrow、DataFusion、Doris PMC 成员），GitHub：https://github.com/jackwener/opencli，官网：https://opencli.info/。思路跟前面几个完全不一样——不操作界面，直接抓背后的 API。

这个方案的好处是根本不跟 UI 较劲。网页怎么改版都无所谓，因为调的是接口不是按钮。效率也高，截图-识别-点击-等待这套流程全免了，直接发请求，Token 消耗几乎为零。

另一个有意思的特性是 Session 复用：通过 Chrome 扩展拿到已登录的 Cookie，不需要 API Key 也不需要 OAuth，登录过就能直接用。目前已覆盖 80+ 平台，用 YAML 声明式管道处理简单提取，TypeScript 适配器应对复杂交互。

不过目前的短板也比较明显：录制引擎只覆盖了只读接口（列表、详情），POST 之类的写操作还没法自动捕获 Request Body。而且发现 API 的过程还得靠浏览器抓包，没法完全脱离。

## 怎么选

不用想太多，就看你的任务到底是什么：

- 写代码、调 Bug → 选 **Chrome DevTools MCP**，调试能力最全，贴近底层
- 跑测试、走流程 → 选 **Playwright MCP**，稳定，但 Token 成本要控制
- 日常看看、简单操作 → 选 **Agent Browser**，轻快省 Token
- 深层爬取、多层钻取 → 选 **Browser Use**，能钻，但慢且贵
- 只要数据不要界面 → 选 **OpenCLI**，直达接口，不受 UI 改版影响
