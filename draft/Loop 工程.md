# 别再手写 Prompt 了：一文看懂 2026 年最火的「Loop 工程」

> 如果说 Prompt 工程是教 AI 怎么做一件事，那么 Loop 工程是设计一套系统，让 AI 自己把一件事持续做完。

## 一、先从一个场景说起

想象两个工程师，都接到同一个任务：修复登录模块的 flaky test（偶发性失败的测试）。

**工程师 A** 的做法：打开 Claude Code，敲下「帮我修一下登录的 bug」，AI 给了一版代码，他看了一眼不对，又敲「不对，是 session 泄漏」，AI 再改一版……一个下午，他成了键盘运动员，在「提问—得到回答—再提问」之间反复横跳。

**工程师 B** 的做法：他写了一行命令——

```bash
/goal "修复 test/auth/login.spec.ts 中的 flaky test。
      成功标准：连续运行 5 次测试全部通过，零失败。不要碰其他测试文件。"
```

然后他去喝咖啡了。系统自己读代码、改代码、跑测试、看到报错、再改、再跑……直到条件达成。

A 在**写 Prompt**，B 在**设计 Loop**。

这个差异，就是 2026 年 6 月整个 AI 编程社区在讨论的「**Loop 工程（Loop Engineering，循环工程）**」。

---

## 二、Loop 工程是什么？

### 一句话定义

**Loop 工程是设计、运营和持续改进反馈循环的工程实践，让 AI 编程 Agent 能够自主完成「规划 → 执行 → 观察 → 修复」的多轮迭代，直到任务真正完成。**

用一句更朴素的话：

> Loop 工程把你从「提示 Agent 的人」变成了「设计提示 Agent 的系统」的工程师。

### 它是怎么火起来的

Loop 工程有一个非常清晰的引爆点——**2026 年 6 月，两句话，百万次转发**。

第一句来自 Anthropic Claude Code 的负责人 **Boris Cherny**：

> 「我不再直接提示 Claude 了。我有一套 Loop 在运行，它们负责提示 Claude 并决定下一步做什么。**我的工作是编写 Loop。**」

几天后，开源 AI Agent 项目 OpenClaw（GitHub 历史上涨星最快的仓库之一）的作者 **Peter Steinberger** 发了一条十二个字的推文：

> 「你不应该再手动提示 AI 编程助手了。你应该设计让 Agent 自己提示自己的 Loop。」

随后，Google 工程师 **Addy Osmani** 在 Substack 上发表长文，把这一实践正式命名并系统化。从此，「Loop 工程」作为一个独立的工程学科诞生。

这两句话之所以引发巨震，是因为它们精准命名了一种很多人**已经感受到、却还没说出口**的趋势：**提示词写得好不好，已经不是瓶颈了；瓶颈在于你为 Agent 设计的那整套运行系统。**

### 理解 Loop：内循环与外循环

要真正理解 Loop，得区分两层循环：

| 层级 | 谁在驱动 | 在做什么 |
| --- | --- | --- |
| **内循环**（Agent 自带） | Agent 自己 | 读文件 → 改代码 → 跑测试 → 看报错 → 再改 |
| **外循环**（你来设计） | 你设计的系统 | 发现任务 → 分派 Agent → 验证结果 → 记录状态 → 开启下一轮 |

**Loop 工程工作在内循环的「上一层」。** 你不再坐在 Agent 旁边，为它的每一步敲一条指令；你在设计一套外部系统，让它替你驾驶内循环，而你去做更有判断价值的事。

## 三、它和 Prompt 工程、Context 工程到底什么区别？

这是被问得最多、也最容易混淆的问题。先看 AI 工程的完整演进路径：

```
Prompt：怎么问 AI
   ↓
Context：给 AI 什么信息
   ↓
Harness：如何组织 AI 的能力
   ↓
Loop：如何让 AI 持续创造结果
```

四者不是「谁取代谁」，而是**一层包一层**的关系。

| 工程阶段 | 核心思想 | 关注点 | AI 的能力 | 人的角色 |
| --- | --- | --- | --- | --- |
| **Prompt 工程** | 设计提示词获得更好的输出 | 怎么问问题 | 单轮生成 | 提问者 |
| **Context 工程** | 组织并提供完整背景信息 | 给 AI 什么信息 | 上下文理解 | 信息组织者 |
| **Harness 工程（编排）** | 连接模型、工具、数据形成工作流 | 如何调用能力 | 执行任务 | 系统设计者 |
| **Loop 工程（循环）** | 构建目标驱动的自主闭环系统 | 如何持续完成目标 | 规划→执行→验证→修复→持续运行 | 规则制定者 |

### Prompt 工程 vs Loop 工程：核心区别

把最容易混的两者拎出来细看：

| 维度 | Prompt 工程 | Loop 工程 |
| --- | --- | --- |
| **优化对象** | 你手写的单条指令 | 自动决定「提示什么、何时提示、结果是否可接受」的整套系统 |
| **工作单位** | 一次你手动输入的对话轮次 | 跨越多轮次、自动运行的完整工作流 |
| **成功衡量** | 第一个回复的质量 | 最终输出的结果质量 |
| **失败模式** | 模型给出了一个差劲的回答 | Loop 设计不良：循环太早停止、忽视错误信号、无法验证完成 |
| **对 Agent 的视角** | 你手持的工具 | 你调度的长期运行进程 |

### 一个关键澄清：Prompt 工程没有死

这是 Loop 工程讨论中最常见的误解。**Loop 工程不是替代 Prompt 工程，而是在它之上的一个更高层次。**

> 一个 Loop 是由多个 Prompt 组成的。写得差的 Prompt 放进 Loop 里，只会让糟糕的工作以更快的速度被产出。

完整的层次是：**Prompt 工程（怎么措辞）→ Context 工程（放什么进上下文）→ Loop 工程（谁来决定何时提示、结果是否合格）**。Loop 是屋顶，Prompt 和 Context 是地基，拆了地基，屋顶立不住。

## 四、为什么我们现在需要 Loop 工程？

### 因为瓶颈转移了

AI 编程工具已经走过了三代，每一代的瓶颈都不一样：

| 阶段 | 代表工具 | 工作方式 | 瓶颈 |
| --- | --- | --- | --- |
| **第一代：自动补全** | GitHub Copilot 早期 | 补全当前行或函数，人主导决策 | 只能辅助，不能自主 |
| **第二代：对话式** | ChatGPT、Claude.ai | 问一句答一句，人手动推进每一步 | **人成了瓶颈，速度受限于打字速度** |
| **第三代：Agent 自主循环** | Claude Code、Codex Agent | Agent 自主规划、执行、验证、循环迭代 | **如何设计让 Loop 可靠运行的系统** |

注意第二代的瓶颈：**人**。你问一句、AI 答一句，整个流程的节奏被你的打字速度和思考停顿死死卡住。AI 能在 3 秒内改完代码，却要等你 10 分钟组织下一句提问。

第三代工具打破了「人」这个瓶颈——Agent 自己会读文件、改代码、跑测试、看报错、再改。**那么人的工作变成了什么？**

> 工程师的核心竞争力，从「会写提示词」变成了「会设计 Loop」。

这就是 Loop 工程出现的根本原因：**当工具已经能自主循环，真正稀缺的能力不再是「怎么用好工具」，而是「怎么设计一套让工具自主、可靠、安全地持续运转的系统」。**

### 更直接的原因：它对症下 Agent 的三种通病

Claude Code 团队公开点名过 Agent 最容易翻车的三种情况，Loop 工程恰好对症下药：

- **智能体懒惰（Agentic laziness）**：复杂多步任务做一半就宣布完成。50 项安全审计只处理了 20 项，剩下的它自信地说「已处理」。
  → Loop 用独立验证 Agent 和可机器验证的完成条件，逼它真的做完。
- **自我偏好（Self-preferential bias）**：让 AI 验证自己的产出时它会偏袒自己——「当裁判和当选手不能是同一个人」。
  → Loop 引入第二个、甚至不同模型的 Agent 来审查，专抓第一个骗过自己的地方。
- **目标漂移（Goal drift）**：多轮之后逐渐偏离目标，每次压缩都丢信息——「第 47 轮时，『不要做 X』这条约束悄悄消失了」。
  → Loop 把目标和边界写进磁盘上的状态文件，而不是寄望于 AI 记在脑子里。

> 「没有质量门禁，AI 会漂移；有了质量门禁，AI 会改进。」

### 还要算清一笔账：循环很美好，直到你看到账单

必须说真话，Loop 是有成本的：

- 单 Agent 循环（一个中等编码任务）：约 **5 万～20 万 token**。
- Fleet 循环（编排者 + 3 个专业 Agent）：约 **50 万～200 万 token**。
- 每天定时跑的循环：**每周数百万 token**。
- 「认真做一周 Loop 工程的成本，可能超过大多数人整个月的 AI 预算。」

这也解释了为什么 DeepSeek、Kimi、GLM 这类低成本模型在 2026 年突然关键——**百万级上下文 + 低 token 定价，才让循环在经济上跑得通。** 本质上，Loop 把成本从「人一轮轮盯」的时间成本，转移成了「系统一轮轮跑」的金钱成本。代价不菲，但回报同样惊人：受 Karpathy autoresearch 启发的自进化循环，让 AI 反复迭代训练代码，**两天跑了 700 个实验，找到 20 个有效优化，性能提升 19%**——这是任何人工调试都难以企及的密度。

### 一句话总结为什么需要它

因为第三代 Agent 已经不需要你「一步步教」，它需要的是你给它一个**清晰的目标、可靠的验证信号、安全的边界**——这三样东西的设计，就是 Loop 工程。

## 五、一个 Loop 是怎么运转的？

一个 Agent Loop 由五个首尾相连的阶段构成，不断迭代：

| 阶段                    | 做什么                 | 典型信号来源                 |
| --------------------- | ------------------- | ---------------------- |
| **1. 意图 Intent**      | 定义目标：成功是什么样子？约束是什么？ | 开发者、Issue、CI 报告        |
| **2. 上下文 Context**    | 收集相关代码、文档、报错日志、规范   | 代码库、测试输出、历史对话          |
| **3. 行动 Action**      | 编辑文件、运行命令、调用工具      | Agent 自主执行             |
| **4. 观察 Observation** | 获取测试结果、编译错误、运行时输出   | 测试框架、类型检查、CI、人工 Review |
| **5. 调整 Adjustment**  | 根据观察更新计划，重复循环       | 下一轮内循环                 |

> Loop 的力量不在于任何单独的步骤，而在于**闭环**。测试失败不只是错误消息，它是新的上下文；类型错误不只是阻断，它是「你之前的假设错了」的信号；Review 评论不只是反馈，它是驱动下一步行动的新观察。

**闭环，是 Loop 区别于一次性对话的本质。**

## 六、一个能独立运行的 Loop，需要六大要素

这套要素缺一不可，它们共同支撑 Loop 的「自主运转」。

**① 自动触发器（Automations）—— Loop 的心跳**
没有自动触发，Loop 就只是「你做过一次的操作」。触发器定义了「什么时候、做什么」。比如 Claude Code 的 `/loop` 命令可以设置每天早上 9 点自动跑。

**② 并行隔离（Worktrees）—— 防止多 Agent 互相踩脚**
多个 Agent 同时运行，最容易在文件冲突上出事。Git Worktree 给每个 Agent 一个独立工作目录，共享 Git 历史但改动隔离。*注意：它消除了文件冲突，但消除不了你的审查瓶颈——你能多快批准变更，才是并行 Agent 数量的真正上限。*

**③ 技能文件（Skills）—— 别让 Agent 每次从零猜规范**
每次新对话，Agent 都要从零推断你的项目规范，这是巨大的浪费。Skill 文件把「项目约定、构建步骤、踩过的坑」固化下来，让 Agent 每次会话自动加载。

**④ 连接器（Connectors / MCP）—— 让 Loop 能在真实世界行动**
只能看本地文件系统的 Loop 能做的事很有限。基于 MCP（模型上下文协议）的连接器让 Agent 能读 Issue、查数据库、调用 API、发 Slack——这是「Agent 说这是修复方案」和「Loop 自动开 PR、CI 过了通知频道」之间的核心差距。

**⑤ 子 Agent（Sub-Agents）—— 制作者与检查者分离**
Loop 最重要的架构决策之一：**写代码的 Agent 和检查代码的 Agent 必须分开。** 写了代码的模型评分自己的作业时太宽容。一个独立检查者（甚至用更强的模型），能抓住第一个 Agent 自圆其说忽略的问题。

**⑥ 持久记忆（Memory）—— 让状态比模型活得更久**
模型每次对话之间会完全遗忘。没有外部记忆，每次 Loop 触发都从零开始。解法朴素而有效：**把状态写在文件里（如 TODO.md），文件放进仓库。** 仓库记得，即使模型不记得。

## 七、怎么构建你的第一个 Loop？

记住一条铁律：**不要一上来就搞「全自动多 Agent 自动合并 PR」的复杂 Loop。** 从最小可用版本开始，理解它的行为后再逐步扩展。

### 第一步：从一个窄任务开始

任务越窄，Agent 越清楚哪些文件重要、哪些验证信号相关。

| ❌ 不好的定义 | ✅ 好的定义 |
| --- | --- |
| 优化仪表盘性能 | 将仪表盘首次加载时间减少 30%，方法是推迟非关键图表加载，同时保持现有过滤器正常 |
| 修复 checkout 问题 | 修复 test/checkout/tax.spec.ts 中失败的税额计算测试 |
| 改进设置页面 | 修复账户删除按钮在移动端（375px 宽度）被截断的布局问题 |

### 第二步：明确告诉它怎么算「完成」

Loop 需要知道「成功」长什么样。你知道验证命令，就直接写进去。

```bash
# ❌ 差：没有验证条件，Agent 不知道何时停止
/goal "Fix the auth bug"

# ✅ 好：有具体的验证命令和成功标准
/goal "修复 test/auth/login.spec.ts 中导致 flaky test 的 session 泄漏。
      成功标准：连续运行 'pnpm test test/auth/login.spec.ts' 5 次全部通过，零失败。
      不要碰其他测试文件。"
```

### 第三步：先上保险，别让它「自由发挥」

在 Loop 能自动开 PR 之前，先让它只写文件、不做任何外部操作，你来审查 diff。推荐用这个「**每日 CI 故障分类 Loop**」作为第一个练手项目——纯只读，只写 TODO.md，不开 PR，风险几乎为零：

```bash
/loop "读取昨天的 CI 失败日志和标记为 'bug' 的 GitHub Issue。
      按可能的原因分类，把带优先级的行动项写入 TODO.md。
      不要编辑任何源代码文件，不要开任何 PR。"
      --schedule "0 8 * * 1-5"
```

每天早上你打开 TODO.md，就知道今天该做什么——而不是花半小时翻 CI 日志。

### 第四步：逐步提升自主程度

| 阶段         | Loop 能做的                 | 人做的               |
| ---------- | ------------------------ | ----------------- |
| **1（只读）**  | 发现问题、分类任务、写状态文件          | 审查 TODO.md，手动决定顺序 |
| **2（草稿）**  | 起草修复、跑测试、写入分支            | 审查 diff，手动 push   |
| **3（半自动）** | 开 Draft PR、跑 CI、通知 Slack | 审查 PR，手动点 Merge   |
| **4（全自动）** | 制作者+检查者双 Agent，CI 过了自动合并 | 异常时介入，定期审计        |

## 八、最后，几个必须警惕的陷阱

Loop 让效率起飞，但有三个问题会**随着 Loop 变好而变得更严峻**，不是更容易。

**① 验证仍是你的责任。**
无人值守的 Loop，也是无人值守地制造错误的 Loop。检查者子 Agent 能降低风险，但「通过了验证」是一个声明，不是证明。**人工审查永远不能消失。**

**② 理解债（Comprehension Debt）积累更快。**
Loop 产出代码越快，你实际理解的代码比例就越低。唯一的解药：**读 Loop 产出的代码**，不要因为它跑得顺就停止理解。

**③ 认知投降（Cognitive Surrender）—— 最隐性的危险。**
当 Loop 自动运转时，接受它返回的任何结果是最舒适的选择。于是出现一种讽刺的局面：

> 两个人可以构建完全相同的 Loop，却得到截然相反的结果——一个用它在深度理解的基础上更快推进，另一个用它来回避理解本身。**Loop 不知道区别。你知道。**

**④ 古德哈特定律（Goodhart's Law）——Agent 会针对「验证器」优化，而不是针对你的真实目标。**

当一项指标变成 Loop 的停止条件，它就不再是好指标了。你的 loop 条件是「测试全部通过」？Agent 可能最后不去修 Bug，而是直接**把失败的测试删了**来交差。所以目标定义不能只有「完成标准」，还必须有「不能怎么做」的边界——这正是 Harness 在 Loop 里发挥作用的时刻：给 Loop 加上护栏，而不只是给它一个终点。

## 写在最后

Loop 工程的本质，是一句话的视角转换：

**从「我怎样让 AI 帮我做这件事」，变成「我怎样设计一个系统，让它持续地、可靠地、安全地自己把这类事做完」。**

提示词写得漂亮，是手艺；Loop 设计得稳当，是工程。

2026 年，AI 已经不再缺「会做」的能力，它缺的是一套能让它「持续做对」的运行系统。而设计这套系统——这是属于这一代工程师的新手艺。

> 别再问「我该怎么提示它」，开始问「我该为它设计怎样的 Loop」。
