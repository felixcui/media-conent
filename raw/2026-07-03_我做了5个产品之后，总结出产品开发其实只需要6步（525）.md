# 我做了5个产品之后，总结出产品开发其实只需要6步（5/25）

**作者**: Seven

**来源**: https://mp.weixin.qq.com/s/77OMyYE43PAm16Cirp59dQ

---

## 摘要

作者基于AI开发经验，将产品开发过程总结为“定、生、修、验、发、查”六步循环法。文章重点解析了前两步：“定”即撰写PRD，将想法翻译成AI可执行的任务，建议分模块下达指令以提升准确率；“生”指快速生成不求完美的第一版，标准是核心流程走通且功能精简，同时需警惕AI遗漏需求、缺少交互逻辑或自作主张加功能等常见坑，并推荐了主流技术栈组合。

hi，大家好，我是seven。一个专注用AI赋能品牌的实践者。

这是《VibeCoding实战：不会写代码，也能做产品》系列拆解第5篇。

今天来聊聊，一个产品从「想法」到「公网可访问」，中间到底经过了什么？

很多人以为最难的是写代码。但做了五个产品之后，我发现最难的从来不是代码。AI 能写代码。真正难的是你知不知道该让 AI 做什么、做完了怎么判断对不对、改的时候怎么改到点上。

我把整个过程总结成六个字：定、生、修、验、发、查。

![Image](https://mmbiz.qpic.cn/mmbiz_png/bu76836DI74CzQHNrL2EkD7tB5icOlbOOGorrDCgCJPVJVWSyNJqPUDUR52ft4M7wxlibXy16O0hOdLfDPGdu3eCKaA7cKUIhzgp4icicxhSnI0/640?wx_fmt=png&from=appmsg)

定义需求 → 生成第一版 → 修改迭代 → 验收检查 → 发布上线 → 观察反馈。

这不是理论框架。是我做每一个产品时都在重复的动作。六步走完就是一个完整循环。走完第一遍可能粗糙，但你已经有了一个上线的产品和真实的用户反馈。然后进入第二遍循环，比上一遍好一点。

这篇一步一步讲。

## 第一步：定——把想法翻译成 AI 能执行的任务

你脑子里的画面，AI 看不到。它需要一份文字版的「订单」。

最怕的沟通方式是：「帮我做一个 AI 工具。」这话的信息量接近于零。AI 不知道做给谁、解决什么问题、长什么样、做到什么程度算完成。

写 PRD（产品需求文档）是整个六步里 ROI 最高的动作。你在这份文档上多花十分钟，后面能省几天的返工。在AI 编码时代，写清楚需求比写代码重要一百倍。

![](https://mmbiz.qpic.cn/mmbiz_jpg/bu76836DI75U4olAss3cAkiaVoiccS4EIS3MgibqEB77UJGZcHJwIibrc8K2m95iaK1U8WnsrjMREnvzMSicpNia75m1LMlOwHC5iamjKz3VZ82hRVY/640?wx_fmt=jpeg)

指路→这里我的上一篇文章详细写了：如何把模糊的想法变成AI能听懂的指令

这里还有一个节奏很重要：不要一次性把整个 PRD 扔给 AI。更有效的方式是一次只给一个功能模块，生成完了验证通过再给下一个。一口气吃太多，AI 容易顾此失彼。

## 第二步：生——第一版要快，不要完美

PRD 对齐了，让 AI 动手。

在它动手之前，用一个「最小闭环检查表」再过一遍你的需求：

```
1. 用户从打开到完成，最少几步？（目标：≤5 步）
2. 核心功能是不是只有 1 个？（不是 3 个，是 1 个）
3. 有没有可以砍掉的功能？（登录？设置？深色模式？）
4. 用户第一次打开，能不能不注册就体验？
5. 这个范围能不能一次性让 AI 生成？（太大就继续砍）
```

第一版的标准就三条：核心流程能走通，首屏能看出这是干什么的，功能不超过 3 个。不需要完美的颜色字体间距，不需要注册登录，不需要支付。这一版的任务是“能用”。

### 第一次生成，常见的几个坑

需求文档太长，AI 只做了前半段。 如果你的 PRD 超过一屏，AI 有可能只处理了开头的部分，后面的功能直接没做。解决办法：把 PRD 拆成两到三段分别发，或者在开头加一句“以下是完整需求，请全部实现”。

第一版出来了但核心功能是假的。 页面看起来很完整，有按钮、有表单、有弹窗，但点了以后什么都不发生。这是因为 AI 先做了 UI（骨架和皮肤），还没接逻辑（神经系统）。这时候不要急着改颜色改布局，先跟 AI 说“XX 按钮点了之后应该发生什么”，把核心交互接通。

AI 自作主张加了你没要的东西。 你写的是一个预约页面，AI 给你加了用户登录、深色模式切换，甚至一个博客板块。这就是为什么 第五章的 PRD 里“不做什么”那一栏很重要。遇到这种情况，直接告诉 AI：“删掉登录功能和博客板块，这一版不需要。”

这些问题都正常。第一次生成不完美，不代表工具不行，也不代表你的需求写得不好。改两三轮之后你就会摸到手感。

关于技术栈，如果你不确定选什么，有一个现阶段被验证过最多次的组合：Next.js + Tailwind CSS + shadcn/ui + Supabase。几乎所有主流 vibe coding 工具（Cursor、Claude Code、v0、Bolt、Lovable）都默认支持这个组合，AI 对它的理解最深、生成质量最高。

第一版生成完毕，测试一下核心流程能走通。如果走通了，立刻做一件事：保存一个版本。 如果你用 Lovable 或 Replit，点一下「检查点/Checkpoint」按钮。如果你用 Cursor 或 Claude Code，在终端里运行 git add. && git commit -m "v1 核心流程通过"。

这一步很多人觉得多余。不多余。这是你的「存档点」。后面改崩了，你能一秒回到这里。比「让 AI 帮你改回来」可靠一百倍。从第一版开始，每次功能正常运行后都保存一个版本。这个习惯会救你的命。

## 第三步：修——别让 AI 猜，给它证据

第一版一定有问题。按钮颜色不对、某个流程走到一半断了、有个地方说不上来哪里别扭。

修改的核心原则：精确描述问题，不要让 AI 猜。

### 修改阶段三个容易踩的坑

第一个坑：越改越乱。 你和 AI 在一个对话里反复修改了二十多轮，突然发现 AI 开始“不聪明了”，比如忘记之前的要求、改 A 的时候忽略 B、在你的要求里偷工减料。这不是 AI 变笨了，是上下文窗口满了。

每个 AI 模型都有一个固定的「记忆容量」（上下文窗口），装着你们之间所有的对话、代码、文件内容。随着对话变长，最早的信息会被挤掉，AI 的表现会越来越差。这个是「上下文腐烂」。

解决方法很简单：开一个新对话，把当前需求重新描述一遍。 不要在一个已经很长的对话里继续追加。如果你用 Claude Code，可以用 /compact 命令压缩上下文，或者用 /clear 重新开始。

第二个坑：改崩了还继续改。 AI 改出了一个 Bug，你让它修这个 Bug，修的过程中又引入了两个新 Bug，你再修，越修越深。这就是「死亡循环」（doom loop）。

正确做法：回滚到上一个正常版本，重新描述你要改的东西。 这就是为什么前面说每次功能正常都要保存版本——你需要一个可以随时退回的安全点。在一个已经崩溃的代码基础上继续修补，效率远低于回到干净的起点重来。

第三个坑：不看修改内容就接受。 AI 改完代码后，很多人直接点「接受」。但 AI 可能在修一个按钮的同时动了你没注意的地方。

如果你用 Cursor 或 Claude Code，养成一个习惯：每次修改后看一眼 diff（差异对比）——看看 AI 到底改了哪些文件、改了哪些行。你不需要读懂每一行代码，但至少确认它只改了该改的地方。

## 第四步：验——AI 做完了不等于做对了

AI 越能干，你越需要验收。 因为 AI 生成的速度太快了，问题也来得快，你不检查就是在堆积隐患。

AI 做完了不等于做对了。你必须自己验收。

这一步是整个流程里最容易被低估的。很多人把大部分时间花在写需求和生成代码上，觉得验收只是「看一眼就行」。但我实际做下来，验收和修复占了我整个开发时间的将近一半。

### 验收五步法：

```
1. 走一遍核心流程
   从「打开页面」到「完成核心动作」，一步不跳。能走通吗？

2. 换一个设备
   电脑上做的？用手机打开。排版正常吗？按钮能点吗？

3. 假装自己是新用户
   清掉脑子里的预设。第一次打开，3 秒内能看懂是干什么的吗？

4. 测试失败状态
   表单什么都不填就提交。网络断了刷新。用户看到友好提示还是白屏？

5. 问一个真人
   把链接发给一个人，什么都不解释。他卡在哪里了？第一反应是什么？
```

过关标准就三条：核心流程能走通，手机上也能用，有真人看过。

在验收阶段，还有一个方法能帮你发现盲区：让另一个 AI 来审。

### 我叫它「互审三角色」：

```
角色 1：Writer——负责写代码的 AI，它只管做出来。
角色 2：Reviewer——另一个 AI，专门挑问题，它只管挑毛病。
角色 3：PM——你。你看双方的产出，决定采纳什么、优先改什么。
```

- 最简单的做法：在编辑器里写完代码，把关键部分复制到另一个 AI 对话窗口让它 review。它没有「我刚写的」这个心理包袱，能更客观地找问题。

- 更省力的做法：让写代码的 AI 自己审查自己，在它生成完代码后直接说：「请审查你刚才生成的代码，检查安全漏洞、边缘情况和 Bug。」AI 自审不能替代真人检查，但能在你看之前先过滤掉一些明显的问题。

- 更省时间的方式是在写代码之前就互审：先让 AI 输出完整的开发计划，把计划贴给另一个 AI 让它 review。改计划只需要改几句话，改代码可能要改几十个文件。

如果你的产品涉及用户登录、权限控制、支付、或者存储用户数据，验收时还要多做一件事：专门检查这些部分的安全性。这不是小题大做——下面「发」那一步我会详细说为什么。

## 第五步：发——不上线就没有真实反馈

发布不是终点。发布是反馈的起点。

验收过了，但产品还只活在你的电脑上。不上线就没有真实世界的反馈。

如果你用 Lovable，发布很简单：点右上角的发布按钮，拿到一个公网 URL。

如果你用 Cursor 或 Claude Code，

### 需要走一个「部署四层图」：

```
第一层：代码（你已经有了）
  ↓ 推送到
第二层：GitHub（代码仓库，存代码的地方）
  ↓ 自动触发
第三层：Vercel（部署平台，把代码变成网站）
  ↓ 绑定
第四层：域名（可选，让网站有自己的地址）
```

每一层做一件事，层和层之间用账号关联自动串通。

这一步你一定会遇到一个概念：环境变量。产品里有些信息（数据库密码、AI 模型的 API Key）不能写在代码里，否则谁都能看到。这些信息存在「环境变量」里，本地开发时放在.env.local 文件，上线后放在 Vercel 的设置面板。两边要一模一样，大部分人卡住就是因为本地配了、线上忘了。

### 域名配置

默认你的网站地址是 yourproject.vercel.app 或 xxx.lovable.app。

这能用，但如果你想让产品看起来正式一点，你需要一个自己的域名。

买域名的流程很简单。我用的是 Namecheap，海外比较主流。国内的话阿里云（万网）和腾讯云也可以。搜索你想要的域名，看看有没有被注册，选一个合适的，结账。.com 的域名一年大概几十块到一百多。

买完之后，你需要做一件事：把域名指向你的部署平台。

这一步叫 DNS 配置。DNS 你可以理解成互联网的电话簿，它告诉浏览器：“当用户输入 yourproduct.com 时，应该去找哪台服务器。”

但如果你用 Vercel + 手动配置，流程是这样的：

```
【域名配置清单】

1. 在 Vercel 后台 → Settings → Domains → 添加你的域名
2. Vercel 会告诉你需要配置哪些 DNS 记录
3. 去域名商后台（Namecheap/阿里云等）→ DNS 设置
4. 添加 A Record 和 CNAME Record，填入 Vercel 给的值
5. 等待 DNS 生效（通常几分钟到几小时）
6. 回到 Vercel 确认域名已连接
```

### 安全检查：比你想象的重要十倍

这不是吓你。而是说 AI 生成的代码必须被当作「未经审查的第三方代码」来对待。AI 写的 CRUD 页面大概率没问题，但它写的认证逻辑、权限控制、支付流程，你不检查就上线，是在赌运气。

一个简单的原则：凡是涉及登录、权限、支付、用户数据的代码，不要让 AI 全自动处理。 你可以让 AI 写，但你必须看一遍，或者让另一个 AI 帮你专门审查这部分。

## 第六步：查——上线后看人怎么真实使用

产品上线了，你会很想看数据。但先别急着盯数据面板。

### 先看人，再看数。

上线第一周，最有价值的信息不是来自 Google Analytics，是来自你微信一对一问的那 3 个用户。

问一个问题就够了：「你打开后第一个想做的事是什么？」

这个问题的答案会告诉你两件事：用户有没有看懂你的产品是干什么的，以及他的第一反应和你设计的路径一不一样。

数据当然也要看，但第一周只看三个数：

```
1. 有没有人来？（日访客数）
2. 来了做什么？（热门页面 / 点击最多的按钮）
3. 做完了没有？（核心流程完成率）
```

### 我把它叫「最小观测三件套」

不需要建仪表盘，不需要做漏斗分析。第一周就看这三个。

如果你用 Vercel 部署，它自带 Analytics 可以看基础的访问数据。想再进一步，可以接 Sentry——它能帮你自动捕获线上报错，用户遇到了白屏或者功能崩溃，你不用等用户反馈就能看到。这两个工具都有免费额度，前期够用。

「查」这一步是很多人会跳过的。大部分 vibe coding 教程讲到部署就结束了。但上线不是终点，上线是产品真正开始的地方。

你在「查」里拿到的反馈，直接变成下一轮「定」的输入——用户觉得核心功能不好用，那下一轮就改核心功能；用户根本没找到核心功能，那下一轮就改首页引导。

六步是一个闭环，「查」和「定」之间的连接是整个闭环最关键的一环。

反馈比数据更值钱。数据告诉你「发生了什么」，反馈告诉你「为什么」。一个用户说「我不知道这个按钮是干什么的」，比你看到「按钮点击率只有 2%」有用一百倍，因为它直接告诉你原因。

## 六步是一个循环，不是一条直线

定、生、修、验、发、查。六步走完就是一个循环。

但它不是走一遍就结束的直线，而是螺旋。每走一轮，你对产品的理解更深一层，下一轮的需求文档更精准，AI 生成的第一版更接近你要的。

第一遍粗糙没关系。关键是走完。

我自己五个项目里，没有一个是第一遍就满意的。但每一个都上线了。上线之后拿到的反馈，比我坐在电脑前想十天更有用。

如果你现在就有一个想做的东西，不妨今天就试试。

完成比完美重要。走完第一遍循环的人，已经超过了 90% 停留在「想做一个东西」的人。
