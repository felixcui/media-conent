# 如何构建一个更"好"的知识库？

> 来源：大淘宝技术（架构部） | 2026年6月10日

本文深入探讨构建高质量 RAG 知识库的垂直技术原理与工程实践。文章首先界定知识库作为外部记忆系统的角色，并引入 RAGAS 框架从检索相关性、生成忠实度及答案相关性维度建立评估标准。随后详细拆解离线索引与在线查询流程，重点分析文档切分策略如 Late Chunking 和意图驱动切分，对比稀疏、稠密及混合检索范式，并阐述HyDE等查询增强技术。此外，文章探讨 Cross-Encoder 重排序机制以优化精度，介绍 AutoRAG 自动化优化、 QuIM-RAG 问题倒排索引及 OpenViking 文件系统范式等前沿架构，旨在通过系统性技术选型解决幻觉、召回不准等问题，实现知识库性能的端到端优化。

## 什么是 Agent 构建中的知识库？

### 考古一下，RAG 的起源

RAG（Retrieval-Augmented Generation，检索增强生成） 由 Facebook AI Research（现 Meta AI）于 2020 年首次提出。

| 项目 | 内容 |
|------|------|
| 论文 | *Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks* |
| 作者 | Patrick Lewis, Ethan Perez, Aleksandra Piktus, Fabio Petroni, Vladimir Karpukhin 等 |
| 发表 | NeurIPS 2020 |

论文的核心贡献在于提出了一种将**参数化记忆（Parametric Memory）**与**非参数化记忆（Non-Parametric Memory）**相结合的架构：

- **参数化记忆**：预训练 seq2seq 模型（如 BART）的模型权重
- **非参数化记忆**：Wikipedia 语料的密集向量索引，通过 DPR（Dense Passage Retriever）构建

### 知识库的定义

在 Agent 构建的语境下，**知识库（Knowledge Base）**是一个外部记忆系统，用于存储和检索不在模型参数中的信息。

RAG 基本工作流程：Query → Retriever → Top-K Documents → Context Augmentation → Generator → Response

### 使用知识库可以解决什么问题？

知识库的核心接口是**上传和召回**，不同版本和理论就是召回的内容和排序的区别。

### RAG vs Long Context

选型建议：
- 数据量 < 50K tokens 且更新频率低 → Long Context
- 数据量大、更新频繁、需要精确召回 → RAG
- 混合方案：RAG 粗筛 + Long Context 精读

## 如何评判一个知识库的好坏？

### 评估框架：RAGAS

RAGAS（Retrieval Augmented Generation Assessment）是目前最广泛采用的 RAG 评估框架，核心价值在于**无参考评估（Reference-Free）**。

三个核心维度：
1. **Context Precision** — 检索器将相关文档排在不相关文档之上的能力
2. **Context Recall** — 回答问题所需的信息有多少被成功检索到
3. **Faithfulness** — 生成的答案在事实上是否与检索到的上下文一致（幻觉检测）
4. **Answer Relevance** — 答案是否直接且恰当地回应了问题

RAG 系统的幻觉可细分为三类：上下文幻觉（Contextual Hallucination）、参数化幻觉（Parametric Hallucination）、采样幻觉（Sampling Hallucination）。

## 构建知识库分为几步？

**离线索引阶段**：Load → Split → Embed → Store
**在线查询阶段**：Query → Retrieve → Rerank → Generate

## 如何获得更好的切分（Chunking）？

切分是影响最大但最容易被忽视的环节。

### 常见切分策略

- **固定长度切分**：最简单，按固定 token 数切分
- **递归切分**：先段落→句子→字符，适合通用文档
- **语义切分**：基于语义相似度判断切分边界（成本效益研究表明性能提升有限但计算成本显著增加）

### 进阶切分策略

- **Late Chunking（延迟切分）**：先编码再切分，每个 chunk 的表示融入全局上下文（Jina AI）
- **意图驱动动态切分**：切分边界由"用户可能问什么"决定，检索准确率提升 5%-67%，分块数量减少 40%-60%
- **Small-to-Big**：检索用小粒度，生成时扩展为大粒度

### 切分参数

- Chunk Size：256-1024 tokens
- Chunk Overlap：10%-20%

## 如何获得更好的召回（Retrieval）？

### 检索范式

- **稀疏检索（BM25）**：基于词频统计
- **稠密检索**：Embedding + 向量相似度
- **混合检索**：BM25 + Dense，RRF 融合

### 进阶检索策略

- **层次化检索**：文档级→段落级两阶段
- **HyDE**：先用 LLM 生成假设性答案，再用答案 Embedding 检索
- **Multi-Query**：生成多个查询变体分别检索
- **EAR（Expand, Rerank, and Retrieve）**：领域外 Top-5/20 准确率提升 5-10 个点

## 如何获得更好的 Rerank？

Reranker 采用**交叉编码器（Cross-Encoder）**架构，将 Query 和 Document 拼接后联合编码，捕捉更细粒度的语义交互。

典型流程：向量检索召回 Top-100 → Reranker 精排 → 取 Top-5 送入 LLM

经验值：K=50~100 是常见选择。

## 前沿案例

- **AutoRAG**：自动化 RAG Pipeline 优化框架
- **QuIM-RAG**：问题倒排索引匹配，将 Query-Document 匹配转化为 Query-Query 匹配
- **OpenViking**（字节跳动）：文件系统范式的上下文数据库，适合长时运行的 Agent

## 核心原则

1. **评估先行**：先建立评估体系，再迭代优化
2. **从简单开始**：递归切分 + 混合检索 + 轻量 Reranker
3. **数据驱动**：根据 bad case 分析定位瓶颈
4. **端到端思维**：关注最终生成质量，而非单一环节指标
