AI 时代，一份 PDF 怎样才能发挥最大价值？

现在处理 PDF 的工具很多，但归根结底，主要有两种形态。

第一种是 **远程 RAG 知识库** 。把 PDF 上传到 ima、NotebookLM 等云端平台，就能直接提问、摘要，有的平台还把音频、视频等能力一并做好。它的优势是功能现成；不足是资料需要上传云端，工作流受平台能力限制，也不会自动和 OB 里的本地笔记建立关系。

第二种是 **本地 Markdown** 。把 PDF 转成保存在自己电脑里的 Markdown，涉及隐私和数据安全的文档也可以留在本机，由本地 Agent 读取、修改，并按自己的需要继续开发 Skill。它的优势是隐私可控、自由度高；不足是受上下文窗口限制，音频、视频等能力通常要自己搭建。

这两种方式都可以马上用，也都可以长期用。区别不在使用时间，而在技术形态和控制权： **远程 RAG 提供现成功能，本地 Markdown 提供本地安全与自由定制。**

![图片](https://mmbiz.qpic.cn/sz_mmbiz_jpg/yxxFy7icEGTQeatozNbYQMjH7js9wBoCeibal8OjGatN2h4KMicnDUvgiaz3COjS0HUPemeZIaxPfZibY0GRE3EDrmPAnR3ZaLv1EzkicQSpd8ZBk/640?from=appmsg)

今天这篇文章只聚焦第二条路线： **你手上已经有一份 PDF，想在本机把它处理成可用的 Markdown，怎么做？**

真正麻烦的是：当 PDF 很多，而且既有普通文档，也有扫描件和复杂报告时，怎样稳定地处理？

今天介绍两个免费开源工具： **MarkItDown 和 MinerU** 。下面讲清楚三件事：为什么要转、两个工具怎样自动分工，以及最后怎样安装。

「PDF 被保存，不等于 PDF 已经能被 Agent 使用。」

01

为什么 PDF 进入知识库前，要先转成 Markdown

![图片](https://mmbiz.qpic.cn/mmbiz_png/yxxFy7icEGTTUpaXgrB5un9XEOiaW1ias9qtfnPrC1KJ7Van1QiaEldttR3FXpNzPN1MedH8e4hzGqX8L0gdX7UTchjgoqhGfFeY94GepSuQ15Q/640?from=appmsg)

上一篇讲 [【OB用户必看】Obsidian 多媒体附件太难管？这个插件一键打通](https://mp.weixin.qq.com/s?__biz=MzkxMTc0MjQ2MA==&mid=2247484652&idx=1&sn=056976d9609d95f27d358500fd39a399&scene=21#wechat_redirect) 时，我介绍过一种做法：图片、音频、视频和 PDF 上传到 OSS，本地只保留轻量的内容。

这解决了大文件的存储问题，却没有真正处理 PDF 里的正文。

PDF 仍然是一份附件。你可以打开它阅读，但不方便直接修改、添加双链，也不方便让本地 Agent 持续搜索和整理其中的内容。

所以，一份 PDF 进入 OB 知识库时，最好同时走两条线：

text

PDF 原件 → OSS 或附件目录：负责保存和回看

PDF 内容 → Markdown：负责搜索、编辑和 AI 处理

**PDF 上传解决的是“文件放在哪里”，转成 Markdown 解决的是“内容怎样被使用”。**

![图片](https://mmbiz.qpic.cn/mmbiz_jpg/yxxFy7icEGTQiaMc3mGsjVTyyrAFzaQq1nnknzM0HFsibKRqdOGpuvIkFOVMggI9PALz3xIhicTyK8PFQDseoOeAZ5GogVj5TdasnWoEnh5v7UE/640?from=appmsg)

Markdown 可以理解为带有简单结构的纯文本。它既能保留标题、列表、表格、图片和链接，又不依赖某一款软件。人能直接阅读，Obsidian 能搜索和双链，Agent 也能稳定地读取、修改并把结果写回文件。

卡帕西的 LLM Wiki 也是这个思路：保留原始资料，再让 Agent 把有价值的内容整理成持续更新的 Markdown 页面。

这里不需要先搭一套复杂的知识管理系统。只要记住一句话： **PDF 转成 Markdown，是让 Agent 真正使用资料的第一步。**

「两个工具不需要人工猜。先快速解析，效果不好再自动换工具。」

02

两个免费工具，怎样自动分工

两个工具的定位很清楚：

工具

适合的 PDF

特点

**MarkItDown**

普通文字型 PDF

轻量、快速，适合批量转换

**MinerU**

扫描件、论文、复杂报告

能处理 OCR、公式、表格和复杂版式

MarkItDown 是微软开源的文件转换工具。除了 PDF，它还支持 Word、PowerPoint、Excel、HTML 和 EPub 等格式。对于正文清楚、排版简单的文件，它通常是更快的选择，结果也比较直接：生成一份可以立即搜索和编辑的 Markdown。

MinerU 更擅长识别页面布局、阅读顺序、图片、表格和公式，也能给扫描件做 OCR。它不仅提取文字，还会把页面里的图片单独抽出来，并生成段落清单等结构化数据。解析能力更强，但安装和运行也更重。

两者的区别，不只是“谁识别得更准”： **MarkItDown 更像快速提取文字，MinerU 更像把整份复杂 PDF 拆成可以重新组织的内容包。**

过去常把 PDF 类型判断交给用户。但在 Agent 工作流里，这一步应该由 Agent 自动完成。

在我的工作流里，这套判断被封装成了一个名为 md-import 的 Agent Skill。它不是第三个转换软件，而是告诉 Agent “什么情况调用哪个工具”的自动化规则：

text

收到 PDF

↓

Agent 抽查前几页和文字层

↓

文字正常、版式简单 → MarkItDown

扫描件或复杂版式 → MinerU

↓

如果快速转换结果过短、为空或乱码，再切到 MinerU

你只需要把 PDF 交给 Agent。Skill 负责指挥，MarkItDown 和 MinerU 负责真正的转换。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_jpg/yxxFy7icEGTTmQSB7VRyYJRNAl8DM4k00EQpO5Xo2DeHUXQEENdWxiaU0GAEGzrsUMcvmV8lPF76EUCBMWyPsfXLTvOBcibI5OomdEiaRYXc0iaM/640?from=appmsg)

「转换不是改一个后缀，而是把 PDF 变成一份真正可用的 Markdown 笔记。」

03

一份 PDF 进入 OB 后，最后会变成什么

先不看 Skill 规则和中间文件，直接看最终结果。

在我的 OB 里，一份 PDF 不再只是一个附件，而是会形成三层内容：

text

第一层：PDF 原件　　 —— 本地可保留，OSS 可查看和下载

第二层：Markdown 全文 —— 负责搜索、双链和 Agent 读取

第三层：资源卡和 Base —— 负责统一管理和查找

如果 PDF 包含大量插图，还会多出一层“图片集合”。先把这三层看懂，后面的自动化就很容易理解。

先说 PDF 原件。Media Manager 把 PDF 上传到 OSS 后，你可以在 OB 里继续保留一份本地原件，断网时也能打开；如果平时不想从本地打开，也可以直接点击 OSS 链接在浏览器中查看，需要时再下载原文件。

这两个入口并不冲突：

text

本地原件 → 可选保留，适合离线回看

OSS 原件 → 可直接在线打开，也可下载

![图片](https://mmbiz.qpic.cn/mmbiz_jpg/yxxFy7icEGTSDtNOCg3tuEHbp0zSF8NAzZC854x9Ge2qmtCpSlK58jPCFYn5gdmOpCP2lkPBmVwBpwDElB9BTic65VTUE5ibWyBL7j92H6icrRI/640?from=appmsg)

### 第一步：先得到一份真正可读的 Markdown

对于普通文字型 PDF，最简单的结果就是：原 PDF 保留，同时在 OB 里多出一份同名 Markdown 笔记。

text

报告.pdf → 原始版式和来源证据

报告.md　　→ 可搜索、可编辑、可被 Agent 读取的正文

Markdown 的输出位置是转换时指定的。不需要把任何人的完整电脑路径写进文章，只要告诉工具：“把结果写进我的 Obsidian 仓库”。

bash

markitdown "<PDF 文件路径>" \\

\-o "<Obsidian 仓库>/<目标目录>/报告.md"

\-o 后面写的是最终.md 文件的位置。在自动化流程里，这个路径会被插件或 Agent 自动补全。

### 第二步：给 PDF 建一张资源卡

只有 Markdown 正文还不够。文件一多，你还需要知道它从哪里来、原件在哪里、是否已经转换。

所以 Media Manager 会为每份 PDF 建立一张资源卡。它不再复制一份 PDF，也不重复放正文，而是这份文档的 **索引资源卡** ，把几个入口串在一起：

text

PDF 资源卡

├── 本地原件：如果选择保留，可直接在 OB 里打开

├── OSS 链接：点击即可查看或下载云端原 PDF

├── Markdown 全文：搜索和 Agent 处理入口

├── 摘要与标签：说明这份 PDF 讲了什么

└── 扫描状态：是普通文档，还是需要 OCR

以后不管 PDF 原件、Markdown 还是摘要放在哪里，都可以先从这张卡进入。即使本地没有打开 PDF，也能在资源卡上直接点击 OSS 链接，查看或下载原文件。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/yxxFy7icEGTQhOZtO8wPrCHK51n0eUwGMsG6FneK9oy5dCiaNLNicAKudqkBQPvHJ56OW7mAueNKVP2EakxULpEYS8Gu3AJPc2CKHXVfvtGhKY/640?from=appmsg)

### 第三步：资源卡自动进入 PDF Base

资源卡的 frontmatter 里有文件名、标签、上传时间、扫描状态和链接等字段。Obsidian Base 会自动读取这些字段，把所有 PDF 整理成一张可筛选的表。

这意味着，你可以直接查看：

\-

最近上传了哪些 PDF；

\-

哪些是普通文档，哪些是扫描件；

\-

哪些还没有整理标签；

\-

哪些已经生成摘要，并从资源卡进入 Markdown 全文。

Base 不需要你每次手工添加一行。新的 PDF 资源卡生成后，它就会自动出现在对应视图里。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/yxxFy7icEGTQUNAgC9msmW2ADbPHN1saVI898XLTK7N6HHTDwmXkX68YSI8kSMIwlPPs3ibVKucoZUWZJGEicctcicdibUX1c0BpwGTfR10q7PnM/640?from=appmsg)

### 第四步：复杂 PDF 里抽出的图片，怎样和原文对应

复杂 PDF 的难点不只是 OCR。一份研报、教材或扫描书可能有几十张、甚至几百张图片。如果只提取文字，图表、插图和页面证据就会丢失。

MinerU 的价值，是它会同时做两件事：

\-

把页面上的文字、标题和阅读顺序整理成 Markdown；

\-

把 PDF 里的原始插图单独抽出，并保留它在正文中出现的位置。

在我的流程里，这些图片不全部复制进 OB vault，而是按文档分组上传到自己的 OSS：

text

pdf-images/

└── <文档名>/

　　├── 图片-001.jpg

　　├── 图片-002.jpg

　　└── ……

随后，后处理会把 Markdown 里的本地图片引用替换成 OSS 地址。打开转换后的 Markdown，图片仍然出现在原来的段落和标题之间。

这一点很重要： **图片不是被丢进一个没有上下文的图库，而是继续挂在它原来所属的文档和章节里。**

系统还会为整份 PDF 建立一张“图集资源卡”，记录：

text

来源 PDF ↔ 转换后的 Markdown ↔ 这份文档的图片集合

图集卡里可以保留文档名、图片数量、OSS 图片目录、来源 PDF 和对应 Markdown。这样做不会为几百张图片建几百张卡，但仍然可以从任意一端追溯回原文。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_jpg/yxxFy7icEGTQrKWTwndZQQRsZo4B8VS0SMo3sOTL61lpVZ7aDZQGRzibOVpPViaU74BqPw97pnY959SfRDtChIDL6pOTGIVNodlmUZL2CMXicLo/640?from=appmsg)

### 这样保存后，图片能不能检索？

可以检索，但要区分两个层次。

现在这套轻量方案已经可以做到：

\-

在 Markdown 里搜索关键词，找到对应段落和旁边的原图；

\-

在 PDF Base 里按文档名、标签、扫描状态找到资源卡；

\-

通过图集资源卡，找到某份 PDF 抽出的全部图片；

\-

从任意一张图所在的 Markdown 位置，追溯它属于哪份 PDF、哪个章节。

但这还不等于“每张图都已经可以按画面含义独立搜索”。如果以后希望直接搜“有趋势线的图”或“某个人物图”，还需要再给单张图片增加 OCR、图片描述或视觉向量索引。

所以，现在完成的是 **来源关联和上下文检索** ；单图语义检索，是可以继续升级的下一层。

04

Media Manager 和 \`md-import\` Skill，到底有什么区别

它们处理的是同一份 PDF，但职责不同： **Media Manager 管理原件，\`md-import\` Skill 处理内容。**

text

Media Manager → PDF 原件上传 OSS + 生成资源卡

md-import Skill → PDF 内容转成 Markdown + 导入 OB

Media Manager 解决“原文件放在哪里、怎样查看和下载”；md-import Skill 解决“PDF 里的内容怎样变成可搜索、可编辑、可被 Agent 读取的 Markdown”。至于底层具体调用哪个转换工具，在这一节不需要再展开。

### 转换完成后，检查这四件事

转换完成后，先检查四件事：

\-

**标题和段落** ：阅读顺序是否正常；

\-

**正文** ：随机对照原 PDF 的两三页，有没有漏字和乱码；

\-

**图片与表格** ：引用能否打开，关键数据有没有错位；

\-

**来源关联** ：PDF 资源卡、Markdown 和图集卡能否互相跳转。

**原件负责保真，Markdown 负责让内容流动，资源卡负责让文档和图片始终找得回来。**

「先跑通一份普通 PDF，再用一份多图扫描件验证 MinerU 接力是否完整。」

05

动手安装吧

### 安装 MarkItDown

MarkItDown 需要 Python 3.10 或更高版本。官方建议在虚拟环境中安装，避免不同项目的依赖互相影响。

bash

pip install 'markitdown\[all\]'

转换一份普通 PDF：

bash

markitdown "$HOME/Downloads/报告.pdf" \\

\-o "$HOME/Downloads/报告.md"

\-o 后面写的是具体的 Markdown 文件路径，而不是文件夹。

![图片](https://mmbiz.qpic.cn/mmbiz_jpg/yxxFy7icEGTRaPFCHVghxJD2tVWMYEUwbxkRyJzeds21YCtwLw8ibb7sk6394MTjo8QBhC4lUa3icY2MFUNoUZgdMylLnTibpkCSUSzic4wxWo8I/640?from=appmsg)

### 安装 MinerU

MinerU 本地安装更重。官方建议先用在线版测试一份不含隐私的样本，确认解析效果后再部署。

bash

pip install --upgrade pip

pip install uv

uv pip install -U "mineru\[all\]"

安装后，可以指定输入 PDF 和输出目录：

bash

mineru -p "$HOME/Downloads/报告.pdf" \\

\-o "$HOME/Downloads/mineru-output"

如果电脑没有合适的 GPU，可以指定纯 CPU 的 pipeline 后端：

bash

mineru -p "$HOME/Downloads/报告.pdf" \\

\-o "$HOME/Downloads/mineru-output" \\

\-b pipeline

涉及合同、病历、内部报告等敏感文件时，不要上传在线服务，直接使用本机方案。

![图片](https://mmbiz.qpic.cn/sz_mmbiz_jpg/yxxFy7icEGTTOm45ECg7el7n9SKFopyqfFSKdibxLhSLrNbyOcr99ms83ibSpibMk1G3kgWRqNUpyd3X4x0P997bo0sWBHFL4tlXKQmQltiaGtK8/640?from=appmsg)

装好以后，就不必再自己判断 PDF 类型。把文件交给 Agent，由它先检查、再选择 MarkItDown 或 MinerU，最后把结果写进 OB。
