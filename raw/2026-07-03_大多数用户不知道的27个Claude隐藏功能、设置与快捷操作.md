**开始之前：先换一个真正有效的心智模型**

大多数开发者把 Claude Code 当成自动售货机。

放进去一段提示词，拿出来一段代码。

正确的心智模型是：Claude Code 是一位极有天赋的初级开发者，但它需要清晰的入职说明、明确的约束，以及定期校准。

给它结构，它就能稳定推进。跳过结构，它就会跑偏。

下面所有技巧，本质上都指向同一件事：

**给 Claude 更好的结构，让它在不依赖你的情况下走得更远。**

## **入门技巧** *先掌握这些。它们能避免 80% 的无效会话。*

**1. 做任何事之前先运行 /init**

每个新项目里的第一件事：运行 /init。

Claude 会读取你的代码库，并生成一个 CLAUDE.md 文件。

这个文件会成为项目大脑，记录架构决策、约定、模式和命名规则。

之后的每一次会话，都会从已经加载好的上下文开始。

没有它：Claude 会猜你的约定。有了它：Claude 已经知道这些约定。

```Bash
/init
```

一个命令就够。完成。不要跳过。

---

**2. 用 /statusline 看清当前状态**

配置 /statusline，把终端变成实时仪表盘。

你可以实时看到 token 使用量、会话指标、模型信息。

```Bash
/statusline
```

不要再凭感觉判断已经消耗了多少上下文。

看数字。

---

**3. 启用语音输入**

免手动编码能让你一直停留在解决问题的状态。

不用停下来打字，你可以直接说出下一条指令。

当你正在审查输出、想要调整方向但不想打断思路时，这尤其有用。

在设置里启用一次，之后持续使用。

---

**4. 用 /context 找出是谁在消耗 token**

你的上下文窗口是一份预算。

当它变重时，用 /context 检查到底是什么在占用它。

```Bash
/context
```

你几乎总会发现一些无效负担：旧文件、无关上下文、过期指令。

把它们清掉。输出质量会立刻变好。

---

**5. 60% 时压缩，不同任务之间清空**

上下文管理有两条规则：

→ 当上下文达到 60% 时运行 /compact：Claude 会压缩对话，同时保留关键信息→ 在完全无关的任务之间运行 /clear：重新开始

```Bash
/compact   # 一个任务拖得比较长时使用
/clear     # 切换到新的无关任务时使用

```

大多数开发者会等到上下文彻底塞满。

到那时，Claude 的表现其实已经在下降了。

60% 是更合适的节点。

![](https://feishu.cn/file/Tf88bXGE4olFyIxANegcKWUcnPf)

---

**6. 始终从计划模式开始**

在 Claude 写下任何一行代码之前，先让它制定计划。

```Bash
# 开始会话时：
"在做任何事之前，先列出方案。
写出步骤。标出所有假设。
如果有任何不清楚的地方，先问我。"

```

为什么有效：

→ 迫使 Claude 先思考再行动→ 提前暴露不一致的假设→ 在消耗计算资源之前，给你一个校准点

以计划模式开始的会话，会多花 3 分钟。

不做计划就开始的会话，往往要多花 3 小时补救。

---

**7. 描述问题，不要只下命令**

大多数人会说：“帮我做一个登录页。”

更好的说法是：“用户使用带特殊字符的密码时无法登录。这是报错。这是约束：我们不能修改数据库结构。”

问题 + 约束 + 上下文 = 输出质量大幅提升。

Claude 不是命令执行器。

它是问题解决者。

把问题交给它。

---

**8. 开始之前，让 Claude 先提问**

把这段加到每一个复杂提示词里：

```Plain Text
开始之前，请先提出你需要我澄清的问题，
这样你才能把这件事做好。请一次性列出所有问题。
```

第一版输出几乎从来不是最好的输出。

最好的输出，通常出现在 Claude 理解边界情况之后。

让它先问。你一次性回答。然后看它推进。

---

**9. 把自检写进待办清单**

给 Claude 任务清单时，加入验证步骤：

```Plain Text
Todo:
1. 构建支付表单
2. 添加输入校验
3. ✓ 验证：用空输入、特殊字符和超过 $10,000 的金额测试
4. 添加错误提示
5. ✓ 验证：确认所有错误状态都已处理并且可见

```

Claude 会在宣布完成之前先检查自己的工作。

没有这一步：Claude 在第 2 步就会宣布胜利。有了这一步：Claude 才会真正完成任务。

## **进阶技巧** *基础动作熟练之后，这些技巧会带来真正的效率跃迁。*

**10. 部署子智能体，并行处理任务**

这是最容易被低估的功能之一。

与其让任务按顺序执行，不如把它们拆给多个拥有隔离上下文的子智能体。

```Plain Text
"启动一个子智能体处理数据库结构。
再启动一个处理 API 端点。
你来协调集成。"
```

并行工作。上下文隔离。互不污染。

生产力提升不是线性的，而是乘法级别的。

---

**11. 把可复用提示词保存成自定义技能**

同一套指令只要写第二遍，你就在浪费时间。

把可复用工作流保存成技能文件：

```Markdown
# skills/code-review.md
审查代码时：
1. 先检查安全漏洞
2. 标出所有 N+1 查询
3. 验证每个外部调用的错误处理
4. 检查变量命名是否符合我们的约定（camelCase、描述性命名）
5. 输出格式：先列严重问题，再给建议
```

加载这个技能。Claude 每次都会遵循你的明确标准。

不要再重复写同一套指令。

---

**12. 简单任务使用更轻量的模型**

Haiku 适合：文件重命名、简单重构、样板代码、配置修改。

Sonnet/Opus 适合：架构决策、复杂调试、多文件改动。

```Plain Text
# 在会话设置中：
"为这个任务使用性价比最高的模型。"
```

大多数开发者什么事都用 Opus。

这就像开法拉利去买菜。

让模型匹配任务。

---

**13. 随着项目演进持续更新 CLAUDE.md**

/init 生成的 CLAUDE.md 不是一次性文档。

项目发生变化时，要同步更新它：

```Markdown
# CLAUDE.md

## 当前技术栈
- Next.js 14（App Router，不是 Pages Router）
- Supabase 用于认证和数据库
- Tailwind + shadcn/ui 用于组件

## 严格规则
- 绝不 commit 或 push。由用户处理所有 git 操作。
- 始终使用 TypeScript。不要使用 JS 文件。
- 所有 API 路由都放在 /app/api/

## 已知问题
- 移动端认证会话有时会过期；任何认证相关工作都要检查这一点
```

过期上下文比没有上下文更糟。

过期的 CLAUDE.md 会让 Claude 自信地朝错误方向前进。

---

**14. 在 CLAUDE.md 中链接外部文件，不要全部贴进去**

不要把所有内容都塞进 CLAUDE.md。

引用外部文件：

```Markdown
# CLAUDE.md

## 架构
参见：/docs/architecture.md

## API 约定
参见：/docs/api-standards.md

## 当前迭代
参见：/docs/sprint-23.md
```

Claude 只会加载需要的部分。

你的上下文保持轻量。token 预算也更健康。

---

**15. 会话一跑偏，立刻终止**

一旦发现 Claude 偏离路径，马上停掉。

不要让错误工作继续叠加。

```Plain Text
停下。这不是我要的。

真正的需求是：[清晰重述需求]
约束是：[写出约束]
只围绕这一点重新开始。
```

叠加起来的错误工作，比重新开始更难修。

尽早发现偏移。强力重写提示词。

---

**16. 挑战第一版输出**

不要直接接受结果。

每次输出之后都问：

```Plain Text
这份结果最薄弱的部分是什么？
如果你有更多时间，会改哪里？
有没有更简单的方案值得考虑？
```

Claude 的第一版回答很少是最佳答案。

经过质疑之后，才更容易得到更好的答案。

---

**17. 设置通知钩子**

不要一直盯着终端。

设置钩子，让长任务完成时主动通知你：

```Bash
# 示例：Claude 完成后通知
claude run "build the feature" && \
  osascript -e 'display notification "Claude done" with title "Claude Code"'
```

去做别的事。

它叫你时再回来。

---

**18. 给 Claude 截图，用于布局验证**

Claude 看不到你正在运行的应用。

但它可以分析应用截图。

```Plain Text
"这是当前状态的截图。
移动端按钮对齐有问题。
修复 CSS，使它符合设计规范。"
```

把截图直接贴进会话。

可视化调试会快很多。

## **高阶技巧** *从这里开始，Claude Code 不再像编码助手，而会开始像一支团队。*

**19. 集成 Chrome DevTools**

让 Claude 通过 DevTools 与正在运行的应用交互。

它可以实时读取控制台错误、网络请求和 DOM 状态。

```Plain Text
"在结账页打开 DevTools。
找出支付按钮没有响应的原因。
检查控制台错误和 Network 面板。"
```

调试变成：Claude 看页面，Claude 修页面。

而不是：你复制粘贴错误，Claude 靠猜。

---

**20. 恢复之前的会话**

不要丢掉上一次会话里的上下文。

```Bash
claude --resume 9901b366-c4b6-4d78-89ad-81e964e373e
```

Claude 会打开完全相同的会话：完整对话历史、完整上下文、停在你离开的地方。

在 Claude Code 历史记录中找到你的会话 ID。

把未来还会回来的会话收藏起来。

**21. 使用 Git Worktrees 运行并行会话**

对复杂项目来说，这是最大的生产力解锁点。

在不同分支上同时运行多个 Claude 会话。

```Bash
# 为并行工作创建 worktree
git worktree add ../project-feature-auth feature/auth
git worktree add ../project-feature-payments feature/payments
git worktree add ../project-feature-dashboard feature/dashboard

# 然后在每个目录里运行独立的 Claude 会话
# Claude 会同时处理三条线
# 没有冲突。不用等待。
```

会话 1：开发认证。会话 2：开发支付。会话 3：开发仪表盘。

全部同时运行。

这就是一天交付一周工作量的方式。

**22. 用 API 替代 MCP 服务器**

MCP 服务器会增加额外开销。

对于性能敏感或高频调用的工作，直接调用 API 。

```Plain Text
# 不要这样：通过 MCP 服务器 → Notion
# 而是这样："直接调用 Notion API：api.notion.com/v1/pages
# 这是我的 API 密钥格式和端点结构。"

```

开销更小。消耗的 token 更少。延迟更低。

昂贵的工具调用 → 换成直接调用API → 立刻改善。

**23. 用 /loop 自动化重复任务**

Claude 可以在无人值守的情况下，最多连续 3 天重复执行任务。

```Bash
"在结账页打开 DevTools。
找出支付按钮没有响应的原因。
检查控制台错误和 Network 面板。"
```

Claude 会连续监控、记录、分类，持续数天。

---

**24. 用手机远程控制**

远程管理你本地的 Claude 会话。

```Bash
# 通过 ngrok 或类似工具暴露 Claude 会话
ngrok http 3000

# 从手机浏览器访问
# 通勤时启动构建
# 不打开电脑也能检查进度
```

当你离开桌面、突然想起有件事忘了启动时，这很有用。

打开手机。启动任务。到家前就会完成。

---

**25. 用自然语言查询数据库**

对于一次性分析，完全可以跳过手写 SQL。

```Plain Text
"在结账页打开 DevTools。
找出支付按钮没有响应的原因。
检查控制台错误和 Network 面板。"
```

Claude 会把问题翻译成 SQL、执行查询并返回答案。

不用写 SQL。不用记数据库结构。只需要提出问题，然后拿到答案。

---

**26. 真正困难的问题使用 Ultrathink**

当问题真的复杂时，分配最大 token，并明确告诉 Claude：

```Plain Text
"在结账页打开 DevTools。
找出支付按钮没有响应的原因。
检查控制台错误和 Network 面板。"
```

90% 的任务，快速处理就够了。

剩下 10% 真正困难的任务，要慢下来。

思考预算本来就在那里。用起来。

**27. Opus 4.8 中的 Ultracode 模式**

为了获得最高编码性能：

```Bash
/effort high
```

这会激活 ultracode，也就是 Claude 最高级别的编码模式。

把它留给：

→ 复杂架构决策→ 安全关键实现→ 性能优化→ 多系统重构

它会消耗更多 token。

但在难题上，它能产出明显更好的代码。

**这 27 个技巧背后的模式**

这份列表里的每一个技巧，都属于以下三类之一：

**上下文规范**：CLAUDE.md、/init、/compact、/clear、文件链接。给 Claude 更好的信息，它就会做出更好的决策。

**智能授权**：子智能体、worktrees、并行会话、/loop。当 Claude 可以同时做很多事时，就别再一件一件来。

**强制自动化**：钩子、远程控制、技能、数据库查询。只要一件事你手动做了两次以上，就应该交给 Claude。

那些产出比别人多10倍的开发者并非更聪明。

他们只是更系统。

---

**从哪里开始**

不要试图今天就实践全部 27 个技巧。

**第 1 周：** /init、/compact、计划模式、自检型待办。这四个就足以改变你的会话质量。

**第 2 周：** 自定义技能、更轻量的模型、更新 CLAUDE.md、挑战每一次输出。你会停止重复写同样的指令。

**第 3 周：** 子智能体、worktrees、并行会话。你会开始同时交付多个功能。

**第 4 周及以后：** /loop、Ultrathink、DevTools、远程控制。你会彻底停止盯着终端。

这个系统会复利增长。

每一层都会让下一层更强。

从开头开始。掌握它。再往上走。
