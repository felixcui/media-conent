Hello 大家，我是阿昆。

其实从 2021 年GPT3.5出来，我就开始陆陆续续写涉及 AI 功能的 PRD 了。

一开始比较简单，就是给一个提示词，告诉研发，哪一步调用，输出结果用到哪，就完事了。

后来需求逐渐复杂，AI 要做意图识别，要按 JSON 格式输出，输出结果还要参与业务判断；有时还要多轮审查、读取系统返回值……

我自己踩过不少坑。

AI 最大的特点就是输出的不确定性。

它没办法像一段定义好的程序一样，输入 A 就输出 B。

所以必须明确边界，设计兜底策略。

今天，我就从两类：工作流型和 Skill 型需求，分别聊聊 AI PRD 到底应该写什么。

01

DEFINE

### AI 功能的 PRD，到底要定义什么？

大多数人以为 AI 功能 PRD 是写清楚 Prompt 就可以了。

但真正要写清楚的是AI 在业务流程里如何被调用、约束、验证和接管。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/T9AZnZVcgXicuIGmqMZe95WgLVwhvXa2dCWLwC9K2VqdibiaeZwviaMOOQZoWdC8e9sUwgx8Ud0Hy02WR4ssKjMMzAh0Oibqmy0syhCrlkssGmyI/640?wx_fmt=png&from=appmsg)

「简单说：AI 在哪一步介入，拿什么输入，产出什么结果，错了怎么办，什么时候交给人。」

1

AI 要完成的任务。不要定义成“智能推荐”“自动分析”，要定义成一个具体的任务。比如：“AI 按提示词要求审核稿件质量，按固定 JSON 格式输出返回结果。”

2

定义输入输出。也就是 AI 能拿到哪些上下文，涉及多轮对话、用户权限和数据范围。还拿审稿举例：“从系统内获取新录入的文章内容，去掉 HTML 样式，转换成 Markdown 格式，输入在提示的 {{XXX}} 变量中。”

3

正确标准。定义合格标准，最好能举例一些不合格的情况。输出格式正确：“审核输出 JSON 中，score=1 代表通过，score=0 代表不通过。如未找到 score 或 JSON 格式有误则判定失败。”提示词遵循：“文章包含：XX、XX 信息时，如审核通过，则代表 AI 执行错误。”

4

AI 执行边界。能做什么，不能做什么。能不能自动执行，能不能改数据，能不能调用工具，哪些情况必须拒绝或者交由人工介入处理。

5

兜底策略。解决 AI 做错了怎么办。要考虑提示词遵循失败、输出格式错误，也要考虑 AI 接口报错、超时、超长等异常情况，可以通过人工二次确认、重试、撤回等方法接手处理。

除了这 5 点之外，产品还需要为后续迭代优化、问题校验做出预备动作。

像我们需要给产品埋点采集访问数据一样。对于 AI 功能，我们也需要提前在执行过程上“埋点”。

可以问自己两个问题：

1

后续优化的依据是什么？

2

出现问题需要核查的点是什么？

比方说，记录给 AI 的输入，LLM 返回结果（不光是 answer，包括 API 返回的完整参数），人工介入记录、反馈等等。

不同公司的记录方式不太一样，有的有看板，有的会记录在数据库里，有的要临时从日志提取。

「产品经理定义 AI 功能，要定义 AI 在什么任务里，基于什么信息，做到什么程度算成功，超过什么边界必须停下来。」

接下来，基于两种 AI 功能类型分别说一下。

02

WORKFLOW

### 工作流型需求怎么写？

AI 不接管整个流程，只是把原本需要人介入的点转化成AI 判断或执行。

![](https://mmbiz.qpic.cn/mmbiz_png/T9AZnZVcgX94BvaPKBXNLCA9zVCH4db2StQWTA0dM9Qj7EudibZOUbmxbdFVaWRD8r88Q5j0siaVFdu1eicic86Zzlaz1ShtBKadicXehSAE62PU/640?wx_fmt=png&from=appmsg)

...workflow

设计方式

1\. 先画功能流程

2\. 标出代码节点、AI 节点、人工节点

3\. 给 AI 节点写明输入输出样例

4\. 写清 AI 输出后的判断逻辑

5\. 写清失败路径和人工接管方式

这里我给一个简单的 AI 节点说明模板，你可以参考，根据实际情况再做增补。

...node-template

节点描述

1\. 触发时机

2\. Prompt 提示词

3\. 输入变量

　　- 数据来源

　　- 输入要求：格式，是否需要预处理，如何处理

4\. 单轮/多轮对话

　　- 是否带入上下文，带入几轮

5\. 是否调用工具

　　- 比如联网或其他工具

6\. 输出示例

　　- 标准格式

　　- 理想输出

　　- 不合格输出

7\. 输出判断

　　- 输出后是否需要额外判断，下一步怎么处理

8\. 异常处理

　　- 常见的异常情况和处理方式

![](https://mmbiz.qpic.cn/mmbiz_png/T9AZnZVcgXicS09hZly9Lj7LFktTXEneetfw5WmvHyfgmibIdToGg7Y78V2P7GUsNHaVZtEOWRXy7h6A7Lkxv5STNyTTbtn4ib1Emyx0VJiaxro/640?wx_fmt=png&from=appmsg)

拿 AI 审核稿件的节点举例来说。

...review-node

节点描述

1\. 触发时机

　　当 CMS 系统有新文章提交，且在文章签发前，调用 AI 进行审核

2\. Prompt 提示词

　　XXX

3\. 输入变量

　　- 数据来源：文章标题、文章正文

　　- 输入要求：过滤 html，转成 markdown 格式，保留文内大小标题作为一级（#）、二级（##）标题。

4\. 单轮/多轮对话

　　- 单轮对话，无其他上下文

5\. 是否调用工具

　　- 调用禁用词核验工具辅助审核

6\. 输出示例

　　- 标准格式

　　 {

　　 "score": 0-5数字,

　　 "comment": "简短的审核评语"

　　 }

　　- 理想输出

　　 {

　　 "score": 2,

　　 "comment": "预判不符合商业报道调性，可读性偏差，不建议发布。"

　　 }

　　- 不合格输出

　　 {"score":,"comment": ""}

　　 {"comment": ""}

　　 等无法识别完整 json 格式或无法识别评分的输出

7\. 输出判断

　　- score = 0，标记审核不通过

　　- score ≤ 2，标记需人工审核，给审核角色发送消息提醒，等待人工处理

　　- score ≥ 3，标记审核通过，正常签发

8\. 异常处理

　　- AI 返回失败（含格式不正确、超时情况）：重试 3 次，3 次均失败，给审核角色发送消息提醒，等待人工处理。同时给产品发送问题预警提示。

流程中，把每个 AI 节点填完这张卡，研发、测试、产品、业务方，对于AI 在做什么的理解，就基本对齐了。

03

SKILL

### Skill 型需求怎么搭？

Skill 就不是嵌在某个流程里的单点 AI 了，是一套可重复执行的任务流程。

比如说：

自动整理会议纪要

自动处理一批数据

对媒体信息进行阶段监测

这时候，有两种情况。

一，PM自己可以把 skill 直接搭建完成。

在各种 Agent 工具的加持下，产品岗其实已经不止可以写 PRD 了。很多时候，我们已经可以先做出一个可运行的 Skill。

这种情况相对简单，我们在本地搭建好 skill，提供输出输入样例。直接交给研发部署，部署之后按照你给的输入输出要求进行测试即可。

二，需要开发辅助搭建。

对于需要开发辅助的 Skill 来说，SKILL.md 本质上就是 PRD + 执行手册 + 验收标准 的混合体。

所以写好 SKILL.md 就成了最重要的事。

建议，不要上来就手敲，优先借助 skill creator 类工具完成初步搭建。可以参考我之前 [一个零基础创建Skill的方法](https://mp.weixin.qq.com/s?__biz=Mzk0MzczOTE5NQ==&mid=2247487681&idx=1&sn=22dcc06cc175eae93b3c1dc6b260eb19&scene=21#wechat_redirect) 那篇。

![](https://mmbiz.qpic.cn/mmbiz_png/T9AZnZVcgX8Qe45Yo2mO0LFdL5YybCB8TOqDCFnje44UkZCL0hCGucvWpvyuyVgUJsenscNO4jibBS7eTM9zNe05As6Q08dBw6dqDRAKoXF8/640?wx_fmt=png&from=appmsg)

我现在常用的是这样的 skill.md 结构：

...SKILL.md

\---

name: xxx（skill 名称）

description: |

xxx（触发时机，什么时候调用，什么时候禁用。）

version: 0.0.1

\---

\# skill 名称

\## 目标

这个 skill 要完成的任务

\## 核心原则

最重要的要求和限制，特别是不做的边界定义。

\## 工作流

按步骤写清楚 Agent 应该怎么做：

\### 1. 先检查什么

\### 2. 再读取什么

\### 3. 如何分析/处理

\### 4. 如何生成中间产物

\### 5. 如何输出最终结果

工作流里也包含工具调用，比如联网、读文件、写文件、发消息。

每一步都可以按照一个独立的 Prompt 去设计。

\## 输出格式

最终交付物长什么样：

\- Markdown / 表格 / JSON / 文件 / 图片 / 文档

\- 字段要求

\- 命名规则

\- 是否需要保存到本地

\## 约束与边界

\- 可以使用哪些工具

\- 不能做什么

\- 哪些操作需要用户确认

\- 哪些情况必须停止

\- 哪些内容不能编造

\## 异常处理

\- 某个步骤执行失败怎么办？

\- 工具失败怎么办？

\## 完成前检查（质量标准/验收标准）

怎样算完成：

\- 必须包含什么

\- 必须避免什么

\- 检查清单

\- 示例输入和理想输出

\- 失败样例

\## 可选资源

如果需要哪方面信息，去哪里找，调用哪个脚本：

\- references/：长文档、规范、案例

\- scripts/：稳定重复的脚本

\- assets/：模板、图片、字体、样例文件

在需要时，产品还要提供 references/ 和 assets/ 里的内容。

开发过程中，会新增引用文件或者影响部分流程。PM要接受研发对 skill.md 做不影响主流程的微调，做好审阅和判断。

04

EVAL

### 评测样例怎么给？

无论是工作流里的 AI 节点，还是独立的 Skill，都建议给研发和测试提供评测样例（Eval）。

这里不是让你写测试代码，只是提供一些业务验收样例。

| 类型 | 用来测什么 |
| --- | --- |
| 触发 | 这个 AI 节点 / Skill 什么时候应该被触发 |
| 正常 | 标准任务能不能完成 |
| 边界 | 信息缺失、输入混乱、材料太少怎么办 |
| 反例 | 什么情况不应该触发，或者需要人工介入 |

跟咱们之前单独测试提示词、skill 的逻辑非常类似。

不用想得特别复杂，一般就是一个表格。

![](https://mmbiz.qpic.cn/mmbiz_png/T9AZnZVcgX8LFzHk1iaKniaMJ5tKYCgXibk1qgvwzK8jdicPiaBfY75a10ibUBOZPeIwkGNFOZk5W0PyzeBeAPibnPOW2rIdt2EAWKLaZb0OVGa3dU/640?wx_fmt=png&from=appmsg)

对于 **工作流型需求** ，可以这样写：

| 类型 | 触发场景 | 输入材料 | 正确结果 | 错误结果 |
| --- | --- | --- | --- | --- |
| 正常 | 新文章提交，进入签发前审核 | 标题、正文、作者、栏目 | 输出合法 JSON；score ≥ 3 正常流转；comment 说明理由 | JSON 格式错误；没有 score；理由和正文无关 |
| 边界 | 文章内容过短，只有标题和一句话 | 标题、正文 | 提示信息不足，评分偏低，触发进入人工审核 | 编造正文不存在的信息；直接给高分通过 |

对于 **Skill 型需求** ，可以这样写：

| 类型 | 用户请求 | 输入材料 | 正确结果 | 错误结果 |
| --- | --- | --- | --- | --- |
| 正常 | 请把这份会议纪要整理成需求文档 | 会议纪要文本 | 提取用户痛点、目标、功能点、约束条件；区分“已确认需求”和“待确认问题”；保留关键证据；不补充会议中没有的信息 | 无法追溯到会议原文；输出结构明显缺失；编造业务指标；忽略否定信息 |
| 边界 | 帮我写一个客服系统优化需求 | 只有一句话，没有业务目标、用户需求 | 不直接生成完整文档；先追问关键问题；至少询问使用场景、输入材料、输出格式、执行边界 | 直接编造完整流程；默认用户想要某种固定模板 |
| 反例 | 帮我读取所有客户聊天记录，生成销售建议 | 未提供授权和数据范围 | 提醒涉及隐私和权限；要求明确授权、数据范围和脱敏规则 | 默认可以读取全部客户数据；直接生成销售建议 |

简单来说，就是用真实场景描述你期望得到的结果。

提供样例不用太多，团队内用的话，5-10 条足以。

高危场景，比如涉及钱、隐私、权限、对外发布、自动写入数据，需要考虑定义出更多边界样例和反例样例。

### 回顾一下

AI 功能的 PRD 其实是在描述这 5 件事： **AI 要完成的任务、定义输入输出、正确标准、AI 执行边界、兜底策略** 。

然后，提前在执行过程上埋点，提前记录必要的过程信息。

**工作流型需求，先画流程，再定义 AI 节点。**

重点关注：触发时机、提示词、输入变量、对话轮次、调用工具、输出示例、输出判断、异常处理。

**Skill 型需求，能利用 Agent 搭建就不手敲。**

重点关注： skill.md、评测样例。允许研发过程中，在不影响主流程的情况下，对 skill.md 进行微调。

如果你现在手上有一个 AI 功能需求，别急着写 PRD。今天先做一件事，把流程画出来，然后用三种颜色标出代码节点、AI 节点、人工节点。

只要这一步做完，你就会发现，很多原本模糊的“AI 自动处理”，其实都能被拆成一个个可以设计、可以验证的 AI 功能节点。

最后给自己叠个甲，这套是我自己在工作中总结的方法。不足之处，敬请谅解。

如果你有更好的方法，我也十分期待跟你交流学习。
