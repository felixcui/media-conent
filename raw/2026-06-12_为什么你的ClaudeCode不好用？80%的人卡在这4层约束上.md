# 为什么你的ClaudeCode不好用？80%的人卡在这4层约束上

**作者**: 三栖有话说

**来源**: https://mp.weixin.qq.com/s/27w5A45haKu8CABi0O1b8g?nwr_flag=1#wechat_redirect

## 摘要

Claude Code体验差的根源往往不是模型本身，而是缺乏稳定的工作约束机制。要让其高效协作，关键在于搭建四层约束体系，其中最核心的是前两层。第一层是全局个人规则，通过配置全局CLAUDE.md固定沟通偏好、安全边界与交付标准，解决重复提醒问题；第二层是项目级规则，通过配置项目CLAUDE.md明确目录结构、开发命令与修改禁忌，避免模型盲目猜测。

很多人装完 Claude Code后，会经历一种很熟悉的落差。

别人分享里，它像一个高级工程师：读项目、拆任务、改代码、跑测试、做总结，一路顺滑。轮到自己这里，它却像一个刚入职但没人带的新人：修登录顺手改了无关文件，按项目风格来却新建一套目录，说好先计划却直接开干，最后测试没跑，只留下一句完成了。

这时候最容易冒出来的念头是：是不是 Claude Code不适合我？

我的判断相反。

Claude Code的很多差体验，根源不在模型本身，而在你没有给它搭 harness。这个词可以理解成Agent的工作环境：方向盘、刹车、仪表盘、路标和验收闸门。模型像发动机，发动机再强，开在没有路网的城市里，也只会四处乱窜。

所以别急着装更多Skill，也别急着换模型。先检查一件事：你的项目里，有没有一套让Claude Code稳定工作的四层约束。

## 第一层：个人规则，先定协作习惯

第一层是全局 CLAUDE.md。

它通常放在用户目录下，作用范围覆盖所有项目。这里不要塞某个项目的技术细节，应该写你的长期偏好和协作边界。

比如：默认用中文沟通；动手前先读相关文件；涉及多文件修改时先给计划；危险操作必须先询问；不能为了让代码跑通就注释掉错误；提交前主动说明是否跑过测试。

这一层像公司的基本制度。它不管某个团队用React还是Vue，它管的是做事方式。

很多人每次都在聊天里重复提醒：别乱删文件，别自动push，先看项目再动手，遇到不确定先说风险。问题是，对话会变长，会压缩，会丢细节。没写进稳定上下文的常识，对Agent来说就像没存在过。

如果不知道全局文件怎么写，可以先从下面这份通用模板开始。它适合放在 `~/.claude/CLAUDE.md` ，重点只写长期稳定的协作习惯，不写某个项目的临时细节。

```
123456789101112131415161718192021222324252627282930313233343536# 全局协作规则
 
## 我的默认偏好

- 默认用中文沟通，除非任务明确要求英文
- 先讲结论，再讲关键原因
- 复杂任务先给简短计划，再开始执行
- 不确定时说明假设和风险，不要装作确定

## 工作方式

- 动手前先阅读相关文件和已有约定
- 修改范围要尽量小，不做与任务无关的重构
- 遇到用户已经修改过的文件，先理解现状再继续
- 发现需求含糊时，先提出最关键的问题
- 能直接验证的结果，优先用命令或测试验证

## 安全边界

- 不要删除用户数据、历史记录、配置文件和密钥
- 不要自动执行 git push、部署、发布和破坏性迁移
- 需要大范围改动时，先说明影响范围
- 命令失败时不要硬猜，先读错误信息

## 交付标准

- 说明改了什么
- 说明验证了什么
- 测试不能跑时说明原因
- 标出仍然存在的风险和下一步建议

## 长期记忆维护

- 反复出现的偏好，沉淀到本文件
- 只在单个项目生效的规则，写到项目 CLAUDE.md
- 具体业务规范和长流程，放到项目 docs 目录
```

这份全局模板解决的是协作底座。它不会让Claude Code立刻懂你的每个项目，但会先把危险动作、沟通方式、验证习惯和交付标准压住。

## 第二层：项目规则，让它知道这里怎么干

第二层是项目级CLAUDE.md。

这层最救命，也最容易被忽略。目录怎么放，开发命令是什么，测试命令是什么，接口风格怎么写，禁改文件清单是什么，组件命名按什么习惯来，都应该放在项目根目录的 CLAUDE.md 里。

没有项目规则时，Claude Code会自己推断。它可能看到一个旧文件，就以为那是当前风格；也可能看到一个实验目录，就以为那是正式结构。你以为它在协作，它其实在猜。

项目CLAUDE.md的价值，是把猜测变成按章办事。

举个最常见的场景：你让它修一个登录bug。没有规则时，它可能顺手重构登录页、改鉴权模块、调整接口返回，还补一套自己喜欢的错误处理。写了规则以后，它会先读相关文件，说明修改范围，只改必要路径，最后跑指定测试。

使用感的变化很明显：你不再反复把它拉回正轨，而是让规则在前面替你站岗。

## 第三层：规范文档，复杂流程别塞满主文件

第三层是规范文档。

CLAUDE.md不应该无限膨胀。官方文档也提醒，指令越具体、越简洁、越结构化，Claude越容易稳定遵守；过大的文件会占用上下文，反而降低跟随度。

更好的做法是：CLAUDE.md放总规则，复杂流程沉到docs目录或规则文件里。

比如 API 设计放 `docs/api.md` ，数据库迁移放 `docs/db.md` ，前端组件规范放 `docs/frontend.md` ，发布流程放 `docs/release.md` 。CLAUDE.md 只写清楚触发条件：涉及数据库迁移时，先阅读 `docs/db.md` ；涉及生产配置时，先停止并确认。

这就是harness的味道。它会避免一次性塞满所有信息，只在正确时刻把正确约束送到模型面前。

好用的Claude Code不需要每秒记住所有事，它需要知道该去哪里查、查完按什么标准做、做完用什么方式验收。

## 第四层：记忆复盘，让坑变成护栏

第四层是记忆和复盘。

Claude Code有记忆能力，也支持通过 `/memory` 查看和编辑相关内容。但记忆不是魔法，它更适合沉淀偏好和经验，不适合替代项目规范。

正确用法是：每次踩坑以后，把稳定规则往上沉。

它误删过生成文件，就把生成文件处理规则写进项目 CLAUDE.md。它部署时漏过环境变量，就把部署检查清单写进文档。它总问同一个测试命令，就把命令放进项目规则。它在某个目录里总按错风格，就给这个目录补一条路径规则。

不要把经验留在一次对话里。对话是临时工位，CLAUDE.md和规范文档才是工作制度。

这四层连起来，才是一个能长期运行的 Claude Code harness：个人规则定协作方式，项目规则定当前项目，规范文档承载复杂流程，记忆复盘让系统越用越准。

## 最后记住一句话

Claude Code很强，但它不是免管理工具。

它更像一个执行力很强的新同事。你不给制度，它会按自己的理解干；你给了制度，它会把执行力放大。

所以下次你觉得 Claude Code不好用，先别问模型是不是不行。打开项目根目录，看一眼有没有 CLAUDE.md。再看看全局规则、项目规则、规范文档、记忆复盘这四层是否接上。

判断同类问题也很简单：凡是你反复提醒两次以上的事，都不该继续留在聊天里。把它写进规则，把规则接到任务，把任务接到验收。

先铺路，再让车跑。

没有路网，再好的车也会乱开。
