## 摘要

在AI Agent时代，代码产出量剧增，工程工作的核心难点已从“写代码”转移到“判断是否该信任这些代码”。最新数据显示，AI虽大幅提升了代码产出，但也导致代码流失率和缺陷率激增，审查时间翻倍，甚至出现大量零审查合并现象。因此，代码审查成为当下软件工程中杠杆最大的技能，而AI工具本身也是协助我们高效完成审查、填补产出与交付价值之间缺口的最好帮手。

Coding agent 现在已经非常强大，而且还在快速进步。随之而来的是：工程工作的难点从"写代码"转移到了"判断是否该信任这些代码"。这让代码审查成为当下软件工程中杠杆最大的技能。而你该如何做审查，很大程度上取决于你的处境：一个没有用户的独立开发者和一个维护十年老系统的团队，面对的根本不是同一个问题。

我对 agentic engineering 的乐观程度，比过去任何时候都高。这些 agent 确实好用，每个月都在变强，在一个普通的工作周里，我现在能交付一年前根本不会尝试的东西。这篇文章想画一张地图，标出有趣的工作去了哪里，因为它确实转移了，而大多数团队还没有完全跟上。

代码审查过去之所以运转良好，源于一个速度上的巧合。高级工程师读代码的速度比初级工程师写代码的速度更快，审查自然而然地跟上了节奏，团队也在阅读彼此 diff 的过程中，顺带吸收了系统的整体结构。这一切并非刻意设计。它是一个简单事实的副产品：写代码慢而贵，读代码快而便宜。

这个前提不再成立。一个 agent 能在我读完这段话之前，就产出上千行格式规范、质量尚可的代码。而人类的阅读速度，从我们开始盯屏幕的那天起就没变过。于是瓶颈向下游转移了，移到了唯一没有变快的步骤：一个人确信这个改动是正确的。我不认为这是损失。我倒觉得这是好事。这是当下软件领域最值得精进的领域，也是我今年投入最多注意力的地方。

不过有个好消息，也奠定了后文的基调。那些产出大量代码的工具，同时也是我用来跟上这些代码的最好帮手。在我自己的项目上，包括那些流行的开源项目，我现在会让 Claude Code 或 Codex 对一批待处理的 PR 做初筛和分级，这确实改变了我分配时间的方式。所以这篇文章不是在反对 AI，后面我会具体讲我怎么用这些工具。

这也不是一份数据罗列，更不是又一轮"让模型写代码到底是福音还是灾难"的争论，因为这种非此即彼的争论毫无意义。唯一经得起真实代码库检验的答案是：完全取决于你是谁。一个用 vibe coding 做副项目、最多十几个人用的开发者，和一个要让十年老企业系统再撑一个季度的团队，两个群体几乎没有可以共享的约束。而流传的大多数建议，本质上只是其中一方在教另一方怎么干活。

## 2026 年的数据到底说了什么

AI 带来的生产力提升是真实的，但原始产出严重高估了实际效果：大约四倍的代码量，只换来十分之一的交付价值增量。这两个数字之间的差距就是审查工作，审查之所以成为杠杆点，就在这里。

过去几年这还只是零散经验和口头争论。现在多个组织已经对此做了大规模测量，这些组织之间没有共同利益，在某些情况下还有商业竞争关系，而结论指向一致：AI 大幅提升了产出，同时也拉低了质量和可审查性。

Faros AI <sup>[1]</sup> 对 4000 个团队中的 22000 名开发者进行了观测，追踪了从低度到高度采纳 AI 过程中发生的变化。这是 2026 年 3 月的数据，几乎是目前能找到的最新的。好的一面值得直说：开发者合并了明显更多的 PR，完成了更多工作，人均产出上升了。但报告还有另一面：

- 代码流失率（code churn）上升 861%
- 事故与 PR 的比率上升 242.7%
- 每个开发者的缺陷率从 9% 上升到 54%
- 中位审查时长上升 441.5%，首次审查时间和平均审查时间都大约翻了一倍
- 零审查合并的 PR 上升 31.3%

最后一个数字是我最难忽视的，因为没有人主动选择过这个结果。没有人决定"我们不再审查了"。只是审查者跟不上提交量，代码开始在无人阅读的情况下合并，然后这变成了常态。我反复琢磨的一个细节是：有成熟、规范工程实践的团队受到的冲击和其他团队一样大。好的流程并没有保护他们，因为代码量到达的速度超过了任何流程的设计容量。

有一个贯穿全文需要留意的事项：CodeRabbit 和 Faros 都在这个市场里卖产品，所以他们的表述不是中立的。这不意味着数字是错的，但厂商研究确实值得带着这个背景去读。

CodeRabbit <sup>[2]</sup> 在 2025 年 12 月研究了 470 个开源 PR，其中 320 个有 AI 参与编写，150 个纯人工编写，发现 AI 参与的改动大约多出 1.7 倍的问题：逻辑和正确性问题上升约 75%，安全问题多出 1.5 到 2 倍，可读性问题增加了两倍以上。他们的 AI 总监 David Loker 将这些描述为"可预测、可量化的弱点，组织必须主动应对"。关键词是"可预测"。既然是已知的、可定位的弱点，那其实是好消息：意味着审查流程，无论是人工还是自动化的，都能精准瞄准这些地方。

GitClear <sup>[3]</sup> 也有值得关注的数据。在他们截至 2025 年的生产力数据中，每天使用 AI 的开发者原始产出大约是非使用者的 4 倍，但和自己一年前的产出相比，实际生产力增长只有约 12%。你生成了大约四倍的代码，却只换来大约十分之一的交付价值增量，而一个人类还得审查全部四倍的代码。值得说明的是，GitClear 的 Bill Harding 明确指出，即使是那 12% 也有一部分是选择偏差，因为更强的开发者集中在了 AI 使用组。四倍代码和十分之一增量之间的差距，就是一句话概括的审查问题。

GitHub <sup>[4]</sup> 报告称 Copilot 审查已经累计执行了超过 6000 万次审查，不到一年增长了 10 倍，平台上超过五分之一的审查涉及 agent。这不再是小众实践，而是代码生产的主流方式。

四组数据、四种方法、一个结论。我们把机器速度的产出灌进了一个为人类速度设计的系统。瓶颈没有消失，它转移到了验证环节 <sup>[5]</sup> ，而审查就是还这笔账的地方。

## 每个人在解决不同的问题

一个改动需要多少审查，几乎完全取决于它的影响范围（blast radius），而你读到的大多数建议，都是由在截然不同影响范围内工作的人写的。

上面那些令人警醒的数据，几乎全部来自企业级遥测和不堪重负的开源维护者。如果你恰好处在那个位置，这些数据说的就是你。但如果你是一个人在做一个只有少数人会用的东西，那大部分数据对你并不适用，你也不必因此感到不安。

三个变量决定了你所在的位置：

- **影响范围：** 出了问题会怎样？什么都不会发生，还是用户投诉、资金损失、隐私数据泄露。
- **代码存活时间：** 一个下周可能重写的原型，还是你要维护多年的代码库。
- **需要理解它的人数：** 只有你自己把整个系统装在脑子里，还是一个需要长期共享所有权的团队。

用同一个 diff 过一遍这三个维度，"好的审查"意味着完全不同的东西。

如果你在独自做一个没有用户的全新项目，审查的第二个作用（在团队中传播知识）对你不存在。你就是整个团队。

合理的做法是大力依赖测试和自动化 <sup>[5]</sup> ，审查真正重要的部分，其余的接受轻度处理。当代码可能一个月后就不存在、也没人会在凌晨三点因它被叫起来时，重复和流失的代价要低得多。但有一个陷阱（很多人是通过痛苦的教训学到的）：这只在测试是真实有效的前提下才成立。在没有安全网的情况下跳过审查，并不是消除了工作，而是以更高的代价推迟了它 <sup>[6]</sup> 。

当项目有了用户。这是最危险的中间地带，而这个转变发生时往往没人注意到。审查的缺陷捕获功能突然变得重要了，因为 bug 现在会伤害到真实的人；审查的知识共享功能也启动了，因为不再只有你一个人。团队往往多保留了几个月独立开发时期的习惯，然后一次事后复盘就来了，Faros 的那些数字不再是别人的图表，而是变成了自己仪表盘上的数据。

另一端是拥有老代码库和大量用户的大型组织。在这里，前面每个令人警醒的数字都会实打实地落在你头上。一个没有人理解的改动就是认知债务（comprehension debt <sup>[7]</sup> ），最终会变成某个人的值班事故。审查同时承担着多个职能，而 agent 产出的体量悄无声息地压垮了所有这些职能。Faros 关于成熟团队的发现，正是针对这种场景。

所以重点不是"企业应该谨慎，独立开发者可以放松"，而是审查的目的随你的处境而变化，规则也必须随之变化。把企业级那种严格锁定的、多 agent 协同的、要求提供证据的流水线硬套到一个两人原型上，你增加的只是无意义的负担。对支付系统执行"测试通过就上线"，你建造的是一个顶着绿色对勾的事故制造机。这个领域里大多数糟糕的建议，都是光谱上的某个位置在给另一个位置开药方。

## 审查现在到底是干什么的

审查最初是为了检查作者的思路、捕获缺陷，以及在团队中共享知识。Agent 确实会推理，但这些推理通常在生成 diff 后就被丢弃了，而不是附在代码上。于是审查者不得不去重建一个从未出现在 diff 中的意图。好消息是：这是一个工具问题，而把推理过程记录下来会让审查轻松很多。

这是真正变了的地方，但很多人还没意识到。

当人类写代码时，意图是自带的。推理过程、被权衡又被放弃的替代方案，都在作者脑中，审查就是你在检查那些推理。现代 agent 确实会推理，通常还是可见的，它们会生成思维链、权衡选项、边做边解释。问题是这些推理通常在 diff 生成的那一刻就被丢弃了。它们很少被保留，很少被附在 PR 上，而且即便保留了，那也是 agent 关于"如何实现任务"的推理，而非人类关于"这个任务本身是否正确"的判断。所以审查从"检查摆在你面前的推理"变成了"重建从未被写下来的意图"，后者更难也更慢。而我们却一直在惊讶为什么审查时间上升了 441% <sup>[1]</sup> 。

一篇 2026 年的论文 AI Slop and the Software Commons <sup>[8]</sup> 分析了 Reddit 和 Hacker News 上 15 个讨论帖中的 1154 条帖子，开发者们在其中讨论"AI 垃圾代码"。其中一位开发者的话让我印象很深：审查一个 agent 的 PR 让他成了"有史以来第一个看到这段代码的人类"。

这直接指向了解法。在正常审查中，作者已经理解了这个改动，你是在检查他们的工作。而在 agent 的 PR 中，还没有人重建过"为什么"。审查者是第一个尝试的人。

正如那篇论文所说，审查"并非为恢复缺失的意图而设计"。但好在缺失的意图是可以恢复的。推理确实存在过，只是我们把它丢弃了。让 agent 说明它试图做什么、排除了哪些方案，把这些作为决策日志 <sup>[6]</sup> 记录在 PR 上，重建成本就会大幅降低。这是一个工具层面的问题，迟早会有人解。

以上都不是说"让 AI 审查 AI"就够了。一个训练背景不同的第二个模型确实能捕获真实的缺陷，而且数量不少，这正是你应该运行一个的原因。它无法提供的是人类的判断：这个改动到底该不该做。这个判断留在人身上，而它恰好是这份工作中最有趣的部分，也是最值得保留的。

## 工具确实好用，但未必是因为它们宣传的那些理由

当前的 AI 审查工具确实好用，而且不同工具标记的往往不是同一行代码，所以正确的做法不是挑选最好的那一个，而是同时运行两个架构不同的工具。

专门的 AI 审查工具现在已经相当成熟，我认为你至少应该让你的主要编程 agent ，或者一个专门的审查 agent 来跑一遍所有东西，包括副项目。

CodeRabbit <sup>[9]</sup> 是部署最广泛的，在独立的 Martian 基准测试 <sup>[10]</sup> （2026 年 1-2 月）中 F1 分数最高，精确率约 49%，召回率为领域最佳。Greptile <sup>[11]</sup> 用精确率换取召回率：在一项基准测试中缺陷捕获率约 82%，对比 CodeRabbit 的 44%，代价是更多误报。Anthropic 的 Code Review <sup>[12]</sup> 报告称其发现中被工程师标记为错误的不到 1%，而我认为最值得给管理者看的数字是：它将内部 PR 获得实质性审查的比例从 16% 提升到了 54%。那些过去只被扫一眼就批准的长尾改动，现在至少有东西在认真读了。

我今年看到的最有用的结果不来自任何厂商。一位工程师并行运行了四个审查工具 <sup>[13]</sup> ，CodeRabbit、Sentry Seer、Greptile 和 Cursor BugBot，在三周半的时间里覆盖了 146 个真实 PR 和 679 项发现：

> 在 617 个不同的标记位置中，93.4% 只被四个工具中的一个捕获。6% 被两个工具捕获。几乎没有被三个工具同时捕获的。被全部四个工具同时捕获的：零。

四个工具从未在同一行代码上达成一致。每个工具擅长不同类别的问题：Greptile 在正确性和架构方面几乎没有误报，CodeRabbit 覆盖面最广且支持一键修复，Seer 在生产故障严重性评估上最强。这就是对抗性审查在真实代码库上的验证，而不是停留在论文里的理论。异质性才是关键。同一个模型跑四份，不过是同一个审查者收了四倍的钱，而四个真正不同的审查者能发现任何单个审查者（包括人类）都找不到的问题。

实际操作建议：不要纠结于找到那个唯一最好的工具，不存在这样的工具。在高风险场景下，运行两个有意选择不同特性的工具（上面那个实验将 Greptile 用于日常正确性检查，Seer 用于生产故障严重性评估，两者几乎没有重叠）。如果你是独立开发者，一个好的审查工具加上真实的测试就够了。而且无论厂商怎么说，都要在自己的代码上测量，因为上面的每一个结果都是在特定代码库上得出的，你的代码库会有自己的情况。

## 我们是否应该让 AI 承担更多审查？

机器对你代码的审查量已经超过了你自己的审查量。唯一真正的决策是你是否有意为之，而你保留多少人类参与应该与你的影响范围成正比。

我不断听到一个问题，一年前这还会被视为异端，但现在问的人是资深工程师：机器是否应该承担更多的审查，甚至大部分？我不再认为这是一个愚蠢的问题。

不太舒服但必须面对的事实是：AI 审查确实有效。Anthropic 的发现中被标记为错误的不到 1%，这些工具能捕获人类一眼扫过去就漏掉的缺陷，而且它们不会在一天审查第三十个 PR 时感到疲倦，而那恰恰是人类最不可靠的时候。与此同时，人类明显跟不上了：零审查合并上升了 31%，审查时间上升了三位数百分比。从实际意义上说，机器对代码的审查量已经超过了我们。更准确的问法不是"我们是否应该让 AI 审查更多"，而是"AI 已经在做了，我们是有意识地管理这件事，还是一边假装人类仍然在读所有东西，一边听之任之"。

Loop engineering <sup>[14]</sup> 把这个问题推得更远。循环的前提是：你不再是那个给 agent 下指令的人，而是构建一个给 agent 下指令的系统。这个系统的核心部件是一个裁判：一个决定工作是否完成才放行的 agent。审查者是下一个被有意从内循环中移除的角色。我们花了一年自动化了编写，而循环系统现在正在自动化检查，人类不断被推到更高、更外的位置。"人处在哪个位置"不是空谈，而是你每次搭建一个循环时都在做的决策，不管你有没有意识到。

我目前的看法是：答案可能不是"一个人类读每一行代码"，那个时代过去了。但答案也不是"让循环自我审查然后置之不理"。当一个 agent 写代码、另一个审查、第三个裁定，你得到的是一个由盲区高度重叠的模型组成的封闭循环，尤其当它们来自同一家族时，它们在相同的地方充满信心地达成一致。一个没有任何人类参与的自信"看起来没问题"是借来的信心 <sup>[15]</sup> ：系统的确定性变成了你的确定性，但实际上没人真正理解了什么。循环可以同时非常确定和非常错误，而这时没有人能发现。

所以人类不会离开，只是上移一个层级。你不再逐个审查 diff，而是负责那些没法交给模型的部分。

判断这个改动该不该做（区别于代码写得对不对）。在高影响范围的节点上把关，因为出错代价很高。还有一个不太好谈的：发现那些根本没被提出过的需求。模型审查的是眼前的代码，很少能指出"大家压根没想到要写"的那些东西，这仍然是一个人类才能填的缺口 <sup>[7]</sup> ，我不觉得它会很快被填上。

人在循环中（human in the loop）变成了人在循环上（human on the loop）：抽样、抽查和审计系统，而不是阅读每个 PR，把有限的注意力花在出错真正会造成伤害的地方。

这已经是我在自己项目上的工作方式了，包括那些开源项目，它们现在一天收到的 PR 比我一个晚上能仔细读完的还多。

我让 Claude Code 或 Codex 对一批待处理的 PR 做初筛：大致判断哪些看起来可以安全合并，哪些需要再看看，哪些是真正高风险的。我不会根据结果自动合并，也不会懒散地批准它认为没问题的所有东西。它给我的是分配注意力的方式。

我可以花几分钟确认它认为低风险的改动，然后把真正认真的时间投入到它标记为危险的那些上。关键是，这不是我过去的审查时间被稍微加速了，而是一种完全不同的工作节奏。以我现在面对的量来说，这是审查队列还能转下去的主要原因。

📷 Codex 和 Claude Code 给我做了一次初筛，按风险排序地阅读一批 PR。分级就是帮助。合并决策仍然是我的。

一个更极端的例子来自 Kun Chen，一位前 Meta L8 工程师，现在作为独立开发者每天交付约 40 个 PR，基本不再审查代码 <sup>[16]</sup> （@petergyang 的报道）。我不赞同这个做法，但他是 L8 级别。他同时运行 20 到 30 个 agent ，并将精力转移到了计划上：他预先写详尽的计划， agent 根据这些计划运行数小时，他说计划质量决定了 agent 能无人值守运行多长时间。这就是我前面说的那种转移。值得仔细说明实际发生了什么，因为并不是他停止了验证。意图没有消失，他在计划中亲自写下了它，所以"有史以来第一个看到这段代码的人类"的问题被半解决了：一个人类确实理解了"为什么"，只是在前面而不是在后面。而且他也不是在没有安全网的情况下工作，他建立了一个自动化的审查门禁（他叫它 No Mistakes）在合并前检查代码，当 agent 遇到困难时他会介入处理。人类在代码写出来之前做昂贵的思考，机器在之后做逐行检查。这很可能就是未来的样子。

但他是一个独立开发者，没有大团队，脚下也没有十年老系统里的地雷。让他每天 40 个 PR 不做审查也合理的那些条件，恰恰是大多数读者不具备的。把他的工作流复制到一个面向大量用户交付的团队，你会在自己的仪表盘上重现 Faros 的那些数字。他没有错，只是他站在光谱很远的那一端。

这又回到了光谱的话题。独立开发、没有用户：让 AI 审查几乎全部代码，在 2026 年是一个站得住脚的立场，你不必为此感到内疚。为大量用户维护大型系统：让机器处理初筛、复审和无聊的 90%，但在承重路径上保留一个真正的人类，不要让循环在任何可能伤害到人的事情上完全闭合。你保留多少人类参与是一个可调的旋钮，该看影响范围，不该看内疚感。

## 具体该怎么做

不要以同样的深度审查所有东西。只在出错代价高的地方投入稀缺的人类注意力，让廉价的确定性检查和 AI 审查工具处理其余部分。

核心思路是：让审查投入与出错代价匹配，把廉价的确定性工作尽可能前置，把人类注意力留给只有人类能做的事。

**按风险分级，不按作者分级。** 一个配置改动只需要跑一下 lint 再扫一眼。对核心业务逻辑路径的修改则需要全套：类型检查、测试、两个不同的 AI 审查器、一个负责该系统的人，以及一次安全审查。不要在样板代码上花重审的力气，也不要仅仅因为测试通过就放过一个大改动。分层的方法 <sup>[5]</sup> 到处都一样，变的只是一个 diff 需要通过几层。

**尽早淘汰高成本的长尾。** 对于被 agent PR 淹没的团队来说，最近最有用的发现是 Early-Stage Prediction of Review Effort <sup>[17]</sup> （2026 年 1 月），该研究考察了 33707 个 agent 编写的 PR。Agent 擅长小型、定义明确的改动，约 28% 几乎立即合并，但它们在收到主观反馈时往往"消失"，放弃审查本身所需的来回交流。（一篇 2026 年的配套论文发现审查者放弃占被拒绝的 agent PR 的 38% <sup>[18]</sup> 。）研究者建立了一个"断路器"，在人类查看之前，通过文件类型和补丁大小等廉价信号预测高维护成本的 PR，效果很好。预先给 agent PR 分级，简单的快速放行，不要让人在一个大改动上花一个小时，结果 agent 一收到推回就不干了。

**提高你愿意审查的门槛。** 被淹没时的解决办法不是锁定仓库，而是拒绝审查没有附带证据的改动 <sup>[19]</sup> 。审查前要求：一份说明改动目的的描述、一个不是 3500 行无注释的 diff、测试输出，以及它确实跑过的证据。这是你停止成为"第一个读这段代码的人类"的方式。你把意图重建的工作推回给提交者（在他那里是廉价的），而不是自己扛（在你这里是昂贵的）。

**刻意保持 PR 的小体量。** agent PR 体量偏大，在 Faros 数据中平均大 51% <sup>[1]</sup> ，而审查者参与度是 PR 能否最终合并的最强预测因子之一。一个大到没法审查的 PR 要么被直接拒绝 <sup>[7]</sup> ，要么更糟，走个形式就过了。让你的 agent 产出小提交。一个人类能实际读完的 diff，现在是设计约束，不是客气。

**比审查代码更仔细地审查测试改动。** 这是最需要注意的 agent 失败模式。Agent 改变了行为，然后"修复"测试，把断言改写为匹配新的、有问题的行为。200 个被修改的测试上面挂着绿色对勾没有任何意义，除非你已经确认这些修改是正确的。把任何重写大量测试的 diff 视为警告信号，优先阅读这些。变异测试（mutation testing）在这里就派上用场了：覆盖率告诉你某行代码被执行了，变异测试告诉你如果那行代码是错的，测试是否会发现。

**把 CI 当作不可动摇的墙。** 注意 GitHub 现在提醒审查者注意的那些模式 <sup>[4]</sup> ：移除测试、跳过 lint、降低覆盖率阈值、复制一个已经存在的辅助函数，以及不受信任的输入流入 prompt。最后一项值得强调，因为 agent 构建的功能是prompt 注入 <sup>[20]</sup> 的新来源：如果一个改动把用户可控的文本直接传给 LLM 调用，而没有考虑这段文本可能指示模型做什么，那么这个漏洞不在 diff 中可见，它潜伏在未来到达的数据中。Agent 也会削弱 CI 来让自己通过，不是出于恶意，只是它在寻找通过测试的最低成本路径。确定性检查是流水线中唯一不会被一段自信的文字说服改变判定的部分，所以要保持严格。

**由人类负责合并。** 模型不会被叫去值班，也没法为它交付的东西负责，所以点击合并的人就是负责人。当一个 AI 审查以平静自信的声音说"看起来没问题"时，它在把未必赢得的信心 <sup>[15]</sup> 交给你。把每次 AI 审查当作传感器，不是裁决：是数据，不是决策。

如果你是独立开发且没有用户，分级、测试改动的纪律和 CI 就是你需要的大部分；其余的在有用户之前都是过度开销。如果你是大型组织，以上全部都是基线，而分级和准入门槛是审查流程能否扩展和悄然崩溃之间的区别。

## 对管理团队的人意味着什么

瓶颈不再是写代码的速度，而是一个可信任的人类能多快对审查结果建立信心。以"AI 让我们更快了"为由削减提供这种信心的人，只是把节省的成本转化为未来的事故。

一个团队能多快交付，现在取决于一个可信任的人能多快确认改动是对的。任何把代码生成视为瓶颈、把审查视为零成本的规划，都会悄悄停滞，而速度仪表盘全程保持绿色。

Faros 的报告对此直言不讳：即使产出上升，QA 和审查工作也在上升，所以以"AI 让我们更快了"为由减少工程师人数是危险的，除非你已经先弥合了审查缺口。高级工程师的审查负担（审查时间上升了三位数百分比）最沉重地落在你最承受不起被拖慢的人身上，而对任何只数已合并 PR 的指标来说，这笔负担根本看不见。

开源维护者最先也最严重地撞上了这堵墙。持续涌入的、看着像样实则空洞的贡献 <sup>[8]</sup> 消耗了真实的分级时间，即使提交者并无恶意，而这就是预警信号。处理得好的企业把审查能力当作一种需要测量、保护和有意分配的真实资源，而不是 AI 释放出来的多余产能。

## 写代码变便宜了，理解没有

代码审查并没有因为 agent 的到来而变得不重要，恰恰相反，它变成了核心活动。写代码这件事正在变得越来越容易，每个月都在降价；持久的优势在于你有没有一套能让自己信任代码的系统。

不要在任何方向上接受一刀切的答案。如果你是独立开发且没有用户，那些关于代码流失和重复的企业恐怖故事是未来的风险，不是眼前的急事，所以依赖你的测试，审查重要的部分，同时诚实地承认推迟的工作还欠着。如果你为大量用户维护大型系统，这里每个令人警醒的数字说的都是你，而唯一能立住的是一套分级的、要求证据的、刻意保持异质性的审查流程，并由人类负责合并。

在整个光谱上不变的是底层经济学。我们让写代码变便宜了，而理解仍然和它一直以来一样昂贵。未来几年做得好的团队，不会是产出代码最多的，而是建起了一套自己真正信得过的审查系统的，而且永远不会把"测试通过了"和"有人理解了这东西在干什么以及为什么"混为一谈。

或者，用 @simonw 反复说的话：你的工作是交付你已经证明能工作的代码 <sup>[21]</sup> 。Agent 没有改变这一点。它们让"证明"成了工作的中心，而不再是事后补的步骤。我觉得这笔交换值得。

深入理解一个系统，理解到能为它背书的程度，是软件领域最持久也最有趣的技能。现在正是在这项技能上精进的最佳时机。
