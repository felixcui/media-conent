## 如何与 AI 高效协作，并让能力持续产生复利

原文作者：Eugene Yan · 2026 年 5 月 3 日

原文链接：How to Work and Compound with AI

导读

很多人已经把 AI 放进日常工作，却依然停留在“每次重新解释、每次从头纠正”的阶段。

Eugene Yan 这篇文章提供了一套更成熟的协作框架：把项目资料变成可检索的上下文，把个人判断标准写进配置，把验证接入工作流，再逐步扩大可以交给 AI 的任务范围。

这样做的价值，会随着时间不断累积。今天完成的代码、文档、分析和决策，会成为下一次会话的上下文；今天纠正的一次错误，也会变成一条配置，降低同类问题再次发生的概率。

怎样才能与 AI 高效协作？具体工作流是什么？它如何扩展？我们又该怎样持续改进自己的系统？

理想状态下，这套系统还应该产生复利。每一份完成的产物，包括代码、文档、分析和决策，都会成为下一次会话的上下文；每一次纠正，也会更新配置，减少未来发生同类错误的概率。

我仍在学习。不过，我已经多次回答过这些问题，所以决定把答案写下来。下次再有人问，我可以直接分享这篇文章。

如果你经常使用 AI，下面很多做法可能已经融入你的工作。其背后的原则具有广泛适用性： **提供良好的上下文、把个人品味写进配置、降低验证成本、委派更大的任务，并完成反馈闭环。**

某项实践与你的情况不符时，可以保留原则，再设计适合自己的做法。阅读时也请留意，这些方法同样适用于任何新加入的协作者。它们与我们帮助新同事熟悉环境、建立合作方式的过程十分相似。

INSIGHT 01

## 把上下文当作基础设施

帮助模型顺利找到你的上下文。

例如，我把所有代码放在 ~/src，把所有知识工作放在 ~/vault。后者继续按 projects/、notes/、kb/ 等目录组织。

工作内容经过整理后，模型可以更轻松地通过 grep 或 glob 检索上下文。清晰的目录树也让它更容易定位并复用以前的代码、项目文档和分析，从而改善当前任务的结果。

把模型接入组织内部的上下文。

组织知识通常分散在 Slack、Google Drive、邮件等系统里。Claude Code、Cowork 和 Claude.ai 大多可以通过 MCP <sup>[3]</sup> 接入这些信息。

除此之外，我还会为每个项目维护一份 INDEX.md。它是一份带注释的资料索引，收录相关文档和频道；每个条目都包含链接、负责人，以及一小段说明，告诉模型里面有什么、什么时候应该阅读。

这些注释非常有用。只有一串链接时，模型必须逐个打开，才能判断哪些内容相关，这会浪费时间和上下文窗口。提前完成注释，相当于只做一次筛选工作，再把结果长期保存在索引中。

像培训新员工一样启动每一次新会话。

每个新会话都从一张白纸开始。因此，可以把项目目录里的 CLAUDE.md 当作第一天交给新同事的入职文档。

Claude 扫描了我各个项目的 CLAUDE.md 后发现，这些文件里会解释缩写、项目代号，也会标明同名同事分别是谁。

我还会在 CLAUDE.md 里给出建议阅读顺序。例如，先快速浏览 INDEX.md，再阅读 TODOS.md，最后查看与当前主题有关的笔记。

建立自己的记忆层。

模型默认不会记得上一次会话发生了什么。因此，任何值得长期保留的信息，都应该写入磁盘。

我把记忆层分成两部分：~/vault 保存项目状态、产物和领域知识等事实；~/.claude 以及其中的 CLAUDE.md、skills/、guides/，保存我的偏好、工作流和个人判断标准。

前者提供上下文，后者提供配置。

INSIGHT 02

## 把个人品味写进配置

先从 ~/.claude/CLAUDE.md 开始。

Claude 会在每次会话开始时读取这个文件。我把它看作一份行为契约。

我的 CLAUDE.md 会说明：表达应该多直接、什么时候需要提出异议、如何处理错误，以及应该教我哪些东西。下面是一份精简版：

...xml

<behavior>

\- 表达要直接；当你不同意时请明确提出。如果我的方案存在问题，直接指出。

\- 对某件事没有把握时，说明你不确定，不要自信地猜测。

\- 任务失败后，先调查根因，再尝试重试。

\- 修改范围应严格围绕当前任务，不要顺手格式化或重构无关代码。

...

</behavior>

<teaching>

我经常需要接触新的系统和领域。

当重要术语出现，而我可能还没有真正掌握时，请用 1 至 2 句话解释，然后继续。

格式如下：

\> 💡 后面跟 1 至 2 句话的解释

...

</teaching>

按目录划分作用范围：先全局，再代码仓库，最后具体项目。

适用于所有场景的偏好，例如行为方式、长期目标和教学方式，放在 ~/.claude/CLAUDE.md。某个代码仓库特有的规范，例如 lint、命名和 Pull Request 要求，放在仓库根目录。

项目专属的上下文，例如目录结构和领域知识，则放在项目目录中。

当你在某个子目录启动 Claude Code 时，它会沿目录树向上查找，并加载沿途的每一份 CLAUDE.md。如果模型在会话中进入另一个子目录，也会读取那个目录里的 CLAUDE.md。具体机制可查看官方文档 <sup>[4]</sup> 。

CLAUDE.md 太长时，把内容拆出去。

过长的 CLAUDE.md 会形成上下文成本。每次会话都会加载全部内容，即使当前任务根本用不到。

解决方法是把大块内容重构成按需加载的指南。不要使用 @import，因为它仍然会把内容直接内联进来。可以在 CLAUDE.md 中注明：遇到相关任务时，再读取对应指南。

这样一来，构建评测的会话就可以跳过写作文档的指南。下面是一段示例：

...xml

<guides>

\- 文档、单页说明以及其他写作任务：~/.claude/guides/writing.md

\- 评测构建与报告：~/.claude/guides/evals.md

\- 数据看板：~/.claude/guides/dashboards.md

...

</guides>

每周至少做一次的事情，可以做成 Skill。

Skill 是一份 Markdown 文件，里面包含名称、触发条件和执行步骤，模型会按需加载。你可以把它理解成用 Markdown 写成的工作流，其中也可以包含判断逻辑。

例如，我的 /polish Skill 会先检查产物的 diff。如果产物会生成某项指标，它就运行对应的评测；如果产物会在浏览器中渲染，它就通过 Claude in Chrome 检查页面；两种情况都不满足时，它会直接运行代码，读取输出或错误。

Skill 同时编码了执行步骤，也编码了“什么情况下应该使用哪一步”的判断。我的部分 Skill 包括：

- • /polish：检查 bug、简化代码、验证输出（通过评测、Claude in Chrome 或其他方式），持续迭代到没有严重反馈，再起草 Pull Request。
- • /write：通过提问与我一起确定大纲，启动研究子 Agent，撰写初稿，再让对抗性审稿人提出意见，持续迭代到没有严重反馈。
- • /daily：读取我的日历、Slack、Pull Request、昨天的日志等内容，整理出当天的优先事项。

我倾向于让 SKILL.md 保持短小，只负责工作流和路由。模板、脚本等知识内容放在独立文件中，模型需要时再读取和运行，就像延迟加载的指南一样。

先亲自完成一次任务，再让模型把过程整理成 Skill。

我的大多数 Skill 都是这样建立的。首先，我会在普通会话里，以交互方式完整做一遍任务；然后让模型把刚才的过程整理成 Skill。

接下来，我会用这个 Skill 再处理同一个任务，或一个相似任务。第一次运行几乎总会需要纠正。我会直接在同一会话里给出反馈，让修改过程留在会话记录中。

最后，我再让模型根据这些纠正和反馈更新 Skill。

你也可以提供符合预期的输出示例作为起点，让模型从中提取模式，比如代码组织方式，或者文档的结构与语气。

通过会话记录改进 Skill，尽量少直接修改文件。

第一版 Skill 很少能达到理想效果，因为它往往过度拟合最初那次会话。这很正常。

运行 Skill 后，如果结果需要调整，就在会话中纠正。尽量别直接打开 SKILL.md 修改。会话里的反馈能给模型形成“修改前与修改后”的配对：我们做了什么、我真正想要什么，以及原因是什么。

当输出符合预期后，再让模型把这些反馈合并进 Skill。经过几轮之后，Skill 会逐渐收敛，你几乎无需再手工修改最终产物。

有些任务用不到整套上下文。

做头脑风暴、探索和粗略起草时，我喜欢使用简单模式：

...bash

CLAUDE\_CODE\_SIMPLE=1 claude

在这种模式下，CLAUDE.md 仍然会加载，但钩子、Skill 和大量工具循环组成的 Agent harness 不会启动。

当我只想边想边说、暂时没有交付任务时，这种方式能让我更接近模型本身。

INSIGHT 03

## 用验证换取更高的自主性

把验证前移，在写入时捕获错误。

我会把验证想象成一架梯子。底层步骤成本低、结果确定；越往上，成本越高，也越需要判断力。我们的目标是在尽可能低的层级解决问题。

靠近底层的是修改后钩子。模型刚更新文件，钩子就运行 ruff format 和 ruff check --fix。这个过程具有确定性，也不消耗 token。

再往上，才是测试、评测和 LLM 审查等环节。

— 验证成本越低，模型能够承担的自主性越高

让模型能够轻松验证自己的工作。

给模型提供反馈回路，它才有机会持续改善输出。

如果系统会生成指标，就让模型运行评测并优化指标；如果结果会在浏览器里渲染，就让模型通过 Claude in Chrome 亲自检查；如果两者都不适用，就让模型运行程序并读取错误。

例如，构建 Docker 镜像时，我会让模型执行构建、读取错误、修改 Dockerfile，再次构建。调整 harness 时，模型会运行评测、阅读会话记录并修复失败案例。

构建数据看板时，模型会在 Chrome 中确认 tooltip 能否正常显示、标签有没有重叠，以及页面叙事是否与数据一致。

面对长时间运行的任务，让模型监督模型。

长会话容易随着错误累积而逐渐偏离目标。一种解决方法是启动第二个拥有全新上下文的会话，让它阅读原始需求，以及主会话最近几轮的记录。

我的最小配置只使用两个 tmux 窗格：一个运行主要开发会话，另一个运行结对程序员会话。初始指令和后续提示都会追加到一个共享文件中。

结对程序员会定期启动，对照原始需求检查主会话最近的记录。一旦发现偏差，它就提供反馈，帮助主会话修正方向。

这种监督可以从不同层面展开。

第一类是 **执行漂移** ：模型有没有正确执行当前任务？它通常是局部、战术层面的问题，例如忽略错误、报告了错误指标，或偏离明确规格。

第二类是 **方向漂移** ：模型当前执行的任务，是否仍然符合最初目标？它属于全局、战略层面的问题，常发生在模型误解原始意图，并花费数小时构建了错误的东西时。

执行漂移需要频繁检查，方向漂移可以降低检查频率。

INSIGHT 04

## 通过委派扩大规模

逐步委派更大块的工作。

有些时候，我们会和模型结对编程：任务很短、反馈很快，人始终留在循环中。这种方式适合快速迭代、探索性分析和原型开发。

随着模型能力增强，我们可以逐步委派更大的任务。提前说明意图、约束和成功标准，然后让模型独立执行。

你无法委派一项自己无法验证的工作。因此，扩大委派范围之前，需要先定义成功标准和衡量指标。

工作方式会从“一次给一条指令”，转向“充分制定计划，再让模型端到端执行”。例如：

“针对这些评测套件，为每一套创建隔离容器，并通过冒烟测试确认它们都能成功构建。然后执行完整评测，记录评测指标与会话记录，再让子 Agent 阅读这些记录，确认评测运行正确。每项评测运行 n 次，以计算置信区间。最后生成报告，检查它是否符合报告指南，并通过 Slack 把结果和报告链接发给我。”

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/HR8kIomiaALTeneTPLs3br8codMYyLf4mpLV7PrUedLs5qSZ17trlWe1yz3PcXhS0ldEaNnUCHjLNABNcOO3rXK8f4RZBfml0V2PVQD8LgUk/640?from=appmsg)

— 明确成功标准后，任务委派、反馈与配置更新会形成复利闭环

并行运行多个会话，同时寻找新的瓶颈。

能够委派更大任务后，我们也可以同时运行更多任务。Claude 的统计显示，我通常会并行运行 3 到 6 个会话。

此时，瓶颈已经从亲自完成工作，转移到编写清晰的规格，以及足够快地审查结果，确保流水线继续运转。中间那段具体执行工作正在被逐渐掏空。

如果多个并行会话需要操作同一个代码仓库，可以使用 Git worktree，让每个会话拥有独立的工作副本，避免互相覆盖修改。

让每个会话的状态清晰可见。

同时运行多个会话时，我需要知道它们分别处于什么状态，以及哪一个需要我介入。

在我的 Mac 上，会话完成时，停止钩子会播放提示音。tmux 窗口标题会显示状态符号（⏳ 表示工作中，🟢 表示已完成），再配上一段由 Haiku 生成的短标签，让我知道每个窗格正在处理什么。

Claude Code 的状态栏还会显示上下文用量和当前模式。停止钩子的声音告诉我有任务结束，tmux 标题告诉我具体是哪一个，状态栏则提供更详细的信息。

...text

\# 停止钩子提醒示例

"Stop": \[

　　{

　　"hooks": \[

　　　　{

　　　　"type": "command",

　　　　"command": "if command -v afplay >/dev/null 2>&1; then afplay -v 1.0 /System/Library/Sounds/Glass.aiff; else tput bel; fi"

　　　　}

　　\]

}

离开电脑时也可以查看进度。

Claude Code 的 /remote-control 可以简化这件事。通勤或排队时，我会打开 Claude 应用里的代码标签页，看看有哪些任务正在运行、哪些任务被阻塞。

必要时，我会补充上下文或提供新指令，让停滞的会话继续推进。这样可以避免任务闲置几个小时。

当然，只在确实紧急时这样做。需要专注于现实生活或休息时，就把注意力留在当下。

INSIGHT 05

## 完成反馈闭环

公开地工作，让上下文持续变得丰富。

当工作发生在共享文档、代码仓库和频道中，所有人，包括模型，都更容易检索并利用这些上下文。我们今天分享的内容，会在明天成为组织上下文的一部分。

可以用一个简单问题检验自己的工作：一位新同事仅凭共享资料，能否复现你上周完成的工作？

如果答案是肯定的，说明你正在为组织上下文作出有效贡献；如果答案是否定的，那些宝贵信息仍然只停留在你的脑中。

我在 CLAUDE.md 中加入了一些自动化指令。每当完成一项重要任务，模型就会在工作日志频道发布简短更新，并附上对应产物、Pull Request 或文档的链接。

从会话记录中寻找配置更新。

让模型阅读以往会话记录，找出工作流里的缺口。

我扫描了自己过去大约 2,500 条用户消息，其中相当一部分包含“你还能不能……”“你检查过……吗”“还是错的”等表达。

这些话通常意味着：模型本应主动完成某件事；CLAUDE.md 或 Skill 需要更新；某个验证步骤缺失，或已经失效。

出现次数能告诉我某类纠正有多频繁，会话记录则能还原具体哪里出了问题。这也是我坚持在会话内完成纠正的原因：这些记录随后可以成为下一次更新 CLAUDE.md 或 Skill 的输入。

定期重构和清理配置。

配置不断增长后，规则之间可能重叠，甚至互相冲突。模型忽略某条规则，有时源于另一条规则与它矛盾。

可以通过定期重构来解决。每条规则或偏好都应该拥有唯一的归属位置，少数高优先级指令可以在主 CLAUDE.md 中重复强调。

我还会检查散落在各级目录中的 settings.json，再把它们集中整理回 ~/.claude。

.

随着模型变得更强，具体配置很可能继续变化。但这些原则仍会长期有用： **提供良好的上下文、编码个人品味、降低验证成本、扩大委派范围，并完成反馈闭环。**

我们正在通过一次次反馈，训练一位协作者。仔细想想，这些原则同样适用于我们与人类团队合作的方式。

你可以让模型读取这份 SETUP.txt <sup>[5]</sup> ，协助你把这些方法应用到自己的环境中。
