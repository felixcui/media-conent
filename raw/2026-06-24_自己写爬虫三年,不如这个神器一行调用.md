## 摘要

数据获取是AI应用和自媒体开发的最大瓶颈，自写爬虫维护成本高且易触发风控，商业数据则昂贵且覆盖窄。文章推荐了RedFox API服务，该产品聚合了抖音、小红书、公众号等十余个国内主流平台的公开数据，包揽反爬、代理等底层细节。用户仅需一个API Key即可调通多平台，直接获取结构化的内容与互动数据，有效解决了数据获取难、成本高的问题。

这两年做 AI 应用和自媒体的人多了，工具也好用了。但真动手做一阵子就会发现，决定一个资讯类应用能不能跑起来的，常常是数据，模型反而排在后面。

模型、Prompt、Agent 框架早就同质化了，个人和团队拉不开差距。卡住大多数人的，是更上游那一环：素材和数据从哪来。

## 拿数据，是最难啃的一块

不管做 "行业热点聚合" 这类应用，还是运营公众号、短视频账号，第一步都一样：先得有数据。

做选题，要知道同行的爆款在写什么。做监控，要追竞品的更新、互动、涨粉。做产品，要把多平台的内容、榜单、互动数据喂给模型。

听着简单，真去拿才知道难。公众号、抖音、小红书各有一整套反爬和风控，想同时做到高价值、稳定、低成本，基本不可能。

我自己试过两条路，都不太行。

第一条，自己写爬虫。一开始当成技术活，以为写写就好。做下去才发现维护成本高得离谱：平台结构隔三差五就变，解析规则跟着失效；要养账号、配 IP 代理池，一不小心就触发风控被封；每接一个新平台，数据格式都不一样，解析逻辑得重写一遍。最麻烦的是它要你持续投精力运维，而这点精力本该花在业务上。对个人和小团队来说，这是个无底洞。

退一步说，就算用开源项目来搭，也省不了多少心。去 GitHub 看看就知道了，爬虫框架和浏览器自动化工具常年霸榜。专门抓这几个平台的 MediaCrawler，到今年初攒了四万多 star，做法是用 Playwright 驱动真实浏览器、保留登录态，省掉逆向签名的麻烦；底层的 Playwright、Puppeteer、Scrapy 这些，更是常年挂在热门列表上。一个方向能冒出这么多项目、被这么多人持续盯着改，恰恰说明这事难，大家都在为同一个痛点发愁。开源解决的是 "能不能抓"，没解决 "稳不稳、要不要你天天伺候"。

第二条，买现成的商业数据产品。省心，但贵，门槛高，覆盖窄。很多服务只做单一平台，或者按套餐、包年收费。对 "先验证个想法" 的早期阶段很不友好，产品还没跑通，预算先去了一半。

折腾一圈，我的需求其实很朴素：一个稳定、覆盖广、按用量付费、几行代码能调通的数据接口。

## RedFox 就是干这个的

最近发现一个挺对路的东西，RedFox（红狐数据，https://redfox.hk ），基本就是冲这些痛点来的，分享给有同样需求的人。

它是个专注公开数据 API 的服务，把国内 10+ 主流社媒平台的公开数据聚合成统一接口。登录、签名、反爬、IP 代理这些工程细节它都包了，你直接调接口拿结构化数据。

覆盖的平台目前有抖音、小红书、视频号、公众号、快手、B 站、西瓜视频、微博、今日头条、百家号、知乎等，拿到的是热点、内容、用户、互动、榜单这类公开维度。

公众号值得单独说一句。做过的人都懂，它大概是这堆平台里最难搞的：没有公开列表，没有规整接口，自己抓格外费劲。RedFox 把这块也啃下来了，能按关键词、按账号检索文章并返回结构化数据。对做选题、追竞品爆文的人，这个含金量不低。

## 用 Claude Code 拼一个多平台监控

举个具体的。想同时盯抖音、公众号的热门内容，两个平台数据格式完全不同，自己写爬虫要维护多套账号、代理和解析规则，三天两头失效。

我换了个做法：直接调 RedFox 的 REST 接口，两个平台统一用 `X-API-KEY` 鉴权，一个下午就拼出了一个分领域热榜聚合器。

**接口结构很简单。** 拿公众号爆文接口举例：

```
import requests

API_KEY = "ak_xxxxxxxx"
BASE    = "https://redfox.hk/story/api"

def fetch_gzh_hot(keyword: str, start_date: str, limit: int = 10):
    resp = requests.post(
        f"{BASE}/gzh/search/hotArticle",
        headers={"X-API-KEY": API_KEY},
        json={"keyword": keyword, "startDate": start_date, "endDate": "2026-06-16"},
    )
    arts = resp.json()["data"]["articles"]
    return sorted(arts, key=lambda x: x["totalScore"], reverse=True)[:limit]
```

抖音热榜是同样的模式，换个 endpoint：

```
def fetch_dy_hot(category: str, date: str, limit: int = 10):
    resp = requests.post(
        f"{BASE}/dy/search/likesRank",
        headers={"X-API-KEY": API_KEY},
        json={"type": category, "startTime": date, "endTime": date},
    )
    return resp.json()["data"]["list"][:limit]
```

**拼聚合器只需要一个循环。** 定义好行业和对应的关键词/抖音赛道映射，批量拉取，合并成统一数据结构：

```
INDUSTRIES = [
    {"name": "AI",   "gzh_kw": "大模型,AIGC,DeepSeek", "dy_type": "科学普及"},
    {"name": "财经",  "gzh_kw": "财经商业",             "dy_type": "财富理财"},
    {"name": "科技",  "gzh_kw": "科技数码",             "dy_type": "数码科技"},
]

dashboard = {}
for ind in INDUSTRIES:
    dashboard[ind["name"]] = {
        "wechat": fetch_gzh_hot(ind["gzh_kw"], start_date="2026-06-09"),
        "douyin": fetch_dy_hot(ind["dy_type"], date="2026-06-15"),
    }
```

返回的字段已经是结构化的——标题、作者、发布时间、阅读/点赞/评论/分享数，直接进库或喂给下游模型，不用自己解析一行 HTML。

**最后加一层轻量渲染** ，把 `dashboard` 写成静态 HTML，配个左侧行业导航 + 右侧公众号/抖音切换 tab，不到 200 行 JS 搞定，部署在本地或任意静态托管上都行。配合AI编程工具整个开发过程的时间大幅缩短，和过去工作量比较天壤之别。
