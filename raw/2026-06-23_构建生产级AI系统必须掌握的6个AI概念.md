# 构建生产级 AI 系统必须掌握的 6 个 AI 概念

> 原帖链接：https://x.com/sairahul1/status/2067540315620405543

![](https://feishu.cn/file/SAd2bZeaUobKeMxYKqicDzRInBh)

我曾经看到一笔 200 美元的账单，一夜之间出现在 AWS 账户里。

不是因为系统崩溃了。

而是因为一个智能体在没有停止条件的情况下循环运行了 6 个小时，每一轮都在调用 OpenAI API。

所有监控面板都显示系统很健康。

直到第二天早上账单来了，才有人发现问题。

这就是你在不了解 AI 系统真实工作方式时构建系统会发生的事。

大多数人学习 AI 工程的顺序是反的。

安装一个库。跟着教程走。调用一个 API。让东西跑起来。感觉自己在进步。

然后某个东西以完全说不通的方式坏掉。

他们随机改一些数字，直到问题消失。

这不是工程。这是拿着键盘祈祷。

以下是解决这个问题的6个概念。

## **一句话解释一切**

任何 AI 系统，无论多复杂，本质上都只是：

![](https://feishu.cn/file/IP6Pbn9oyoZrUkxAp0KcRHiNnyg)

**记忆（RAG）+ 思考（LLM + tokens）+ 行动（Agents）+ 度量（Evals）**

……再通过**上下文工程**组装起来。

这就是整个领域。

下面所有内容，只是在拆开解释每一部分到底意味着什么。

---

## **1. token 与上下文窗口**

![](https://feishu.cn/file/Z4twb96uSovkaOxh1mbcD8UDnKh)

LLM 读的不是单词。它读的是被称为 token 的片段。

“engineering” → 1 个 token

“unbelievable” → 2 个 token。空格和标点也会计入。

每个模型都有上下文窗口，也就是一次能容纳的 token 硬上限。

→ Claude：200,000 token

→ GPT-5：400,000 token

可以把它想象成会议室里的白板。

模型只能处理当前写在白板上的东西。

当白板写满时，旧笔记会被擦掉，给新内容腾地方。

模型并没有失去思考能力。

它只是失去了访问早期信息的能力。

**为什么这会破坏生产系统：**

→ token 会产生费用，每次 API 调用都会按输入和输出 token 计费

→ 长聊天历史会很快填满上下文窗口

→ 当上下文满了，早期指令会被静默丢掉

→ 什么内容进入上下文，是工程决策，不是默认设置

**证明这一点的故障：**

一个团队构建了一个客服智能体，每次请求都把完整 12 个月聊天历史放进上下文。

在测试中，只有 5 次交互时效果很好。

到了生产环境，50 次交互之后，智能体开始忽略自己的系统提示词。

指令仍然在那里。

只是被 80,000 个 token 的对话历史埋住了。

模型实际上已经不再关注它们。

修复方法不是换更好的模型。

而是把较旧的历史压缩成摘要，让窗口保持聚焦。

**令人不舒服的真相：**

大多数“提示词工程失败”，其实都是 token 和上下文窗口问题的伪装。

工程师会责怪提示词，但真正的问题是：关键指令藏在 500 行上下文的第 3 行，模型已经不再给它足够权重。

## **2. Embedding 与向量搜索**

![](https://feishu.cn/file/OyqdbW863ooSU3xVgT6cXS6untc)

Embedding 会把含义转成数字，这样“相似”就可以用数学计算。

它解决的问题是：

你有 50,000 份文档。用户问了一个问题。你需要找出最相关的 3 份，但不能每次都读完 50,000 份文档。

关键词搜索在这里会失败。

如果文档里写的是 “automobile”，而用户问的是 “cars”，关键词搜索会漏掉它。

不是因为答案不存在，而是因为词没有匹配上。

Embedding 用另一种方式解决这个问题。

Embedding 模型会把文本转换成向量，也就是一组代表数学空间中语义位置的数字。

语义相似的文本 → 数值上相近的向量。

“car” 和 “automobile” → 距离很近

“car” 和 “photosynthesis” → 距离很远

**向量搜索实际如何工作：**

1. 每份文档都会被转换成向量并存储起来
2. 用户的问题也会被转换成向量
3. 系统找到与问题向量最接近的已存储向量
4. 这些就是最相关的文档

这不是近似的魔法，而是几何。

相似度是真实可计算的数学属性。

**它在生产中出现在哪里：**

→ 任何文档系统里的语义搜索

→ 查找相似产品、文章、用户画像

→ RAG 中的检索步骤，也就是下一个概念

→ AI 智能体的记忆

## **3. RAG（检索增强生成）**

![](https://feishu.cn/file/QAfAbkb7iogImGxohXccZIS7nse)

与其用你的数据训练模型，不如在查询时取回相关数据，并把它作为上下文交给模型。

RAG 解决的问题是：

LLM 知道很多东西，但不知道你的数据。

你公司的内部文档。你的产品数据库。你的客服历史。

这些都不在训练集中。

有两个选择：用你的数据训练模型，昂贵、缓慢，而且很快过期；或者在模型需要时，把你的数据准确地交给它。

RAG 就是第二种选择的系统化做法。

**3 步流水线：**

→ 检索：

问题变成向量 → 向量数据库找到最相似的已存储文档 → 取回前 3 到 5 个片段

→ 增强：

取回的文档被加入模型上下文 → 提示词变成“根据这段上下文，回答这个问题”

→ 生成：

模型基于你的真实数据作答，而不是凭空幻觉

**RAG 会在哪里失效：**

→ 检索差 = 答案差。模型只能使用它收到的内容

→ 糟糕的切块会把答案和上下文拆开

→ 如果检索不到有用内容，模型仍然可能幻觉

**一个真实的 RAG 故障：**

一个团队为一份 500 页技术手册构建了内部知识助手。

演示时效果完美。上线后，答案变得含糊，有时还错误。

问题在于切块大小。

他们按原始字符数，把手册切成 1,000 token 的片段。

表格在行中间被切断。分步骤说明在步骤中间被切断。

检索找到了正确的大致区域，但漏掉了真正的答案。

把切块大小减半，并添加重叠，一夜之间修掉了 80% 的问题。

**一个强硬观点：**

如果检索做得差，RAG 就被高估了。

LLM 无法修复糟糕的检索。它只能围绕糟糕的检索结果编造。

如果你看到错误答案，不要再调提示词了。

开始测量检索精度。

答案在那里。

## **4. 智能体循环**

![](https://feishu.cn/file/D9XdbRWxgoyxhuxvy8vcsuVynwg)

智能体的工作方式，是不断选择一个动作、执行它、观察结果，再决定下一步做什么，直到任务完成。

普通 LLM 调用是无状态的。你提问，它回答，结束。

智能体是有状态的。它行动，观察，决策，重复。

**用普通话说，这个循环是：**

1. 接收目标
2. 决定下一步动作
3. 执行动作：搜索、写代码、读取文件
4. 观察结果
5. 根据学到的东西决定下一步动作
6. 重复，直到目标完成
7. 返回最终答案

工具让智能体拥有能力。

没有工具，LLM 只能回复文本。

有了工具，它就能搜索网页、读取文件、写代码、调用 API、触发你定义的任何动作。

**初学者总会搞错的三件事：**

→ 没有停止条件的智能体会永远运行。你必须定义什么时候停止：步骤上限、时间上限或目标条件

→ 工具越多不等于表现越好。工具太多会让模型不知道该用哪个

→ 工具错误需要显式处理。静默失败会让智能体自信地产出垃圾

**那次一夜 200 美元事故的细节：**

那个智能体没有最大步骤数。它的目标是：研究一个主题并产出摘要。

其中一个网页搜索工具返回了空结果。

智能体不知道该如何停止。

它持续搜索、重试、生成中间摘要，而每个中间摘要又触发下一次搜索。

6 小时后：847 次 LLM 调用，消耗 210 万 token。一份看起来连贯、但完全循环论证的摘要。一张 200 美元账单。

修复只用了三行：最大步骤计数器、空结果的显式处理器，以及低置信度时的升级路径。

同一个智能体现在平均不到 12 次调用就能完成。

**你需要听到的观点：**

大多数智能体失败，不是因为模型差，而是因为工程师把循环当成会自我管理的东西。

它不会。

护栏、停止条件、错误处理器，必须从一开始就内置，而不是第一次事故后再补。

## **5. Evals（评测）**

![](https://feishu.cn/file/W2zIbwKK9ou5fbxgKOjcwSgunNf)

Evals 让你知道 AI 系统是否真的在工作，也让你知道某次改动到底让它变好了还是变坏了。

这是大多数教程会跳过的概念，因为它不够耀眼。

但它也是区分“会做演示的工程师”和“会构建生产系统的工程师”的分界线。

**没有评测会有什么问题：**

你修改了提示词。更新了检索逻辑。换成了更新的模型。

它变好了吗？

你不知道。你可以手动看几个例子，但那只是感觉，不是证据。

**评测实际长什么样：**

→ 黄金数据集：25 到 50 个真实输入，以及已知正确输出，覆盖主要用例，再加 5 个已知困难边界情况

→ 尽可能使用二元指标：

— RAG 系统是否取回了正确文档？是/否

— 智能体是否无错误完成任务？是/否

— 回复是否包含必要信息？是/否

→ 随时间追踪聚合分数：

— 检索准确率：89% → 做了改动 → 84%。发现回退。

— 任务完成率：76% → 新智能体版本 → 81%。确认改进。

**评测循环：**

部署 → 用评测度量 → 找到失败案例 → 把失败案例加入黄金数据集 → 修复 → 再次运行评测 → 比较分数 → 只有数字改善时才发布

**实话实说：**

“有用性：3.7/5”不会告诉你任何可行动信息。

“84% 的时间取回了正确文档”会准确告诉你问题在哪里，以及修复带来了多少改善。

没有评测的 AI 系统不是产品。

它是一个你无法放心修改的演示。

## **6. 上下文工程**

![](https://feishu.cn/file/E4I8bkNEXoI90NxMHwscoT63nNd)

上下文工程是一门决定什么信息进入模型上下文窗口、这些信息如何组织、以及什么应该被排除在外的学科。

下面这个观点会让人不舒服：

上下文工程比提示词工程更重要。

一个普通提示词，如果放在精心整理的上下文里，表现会胜过一个被噪音淹没的绝妙提示词。每一次都是如此。

大多数团队把 80% 的优化精力花在提示词上，几乎不花在上下文上。

结果会反映这一点。

**天真的做法会失败：**

把所有东西都放进去。所有历史。所有检索到的文档。每个工具描述。系统提示词。用户消息。全部塞进去。

这会因为一个稳定原因而失败：模型会分不清什么最重要。

有一个有记录的效应叫“lost in the middle”：埋在长上下文中间的信息，更不容易被模型使用。

**上下文工程实际包含什么：**

→ 选择：这个具体决策需要哪些文档、事实或历史？

→ 压缩：能否把对话较早部分压缩成摘要，节省 token？

→ 排序：关键指令应该放在开头和结尾，而不是中间

→ 修剪：什么内容可以删除，同时不影响输出质量？

→ 结构：标题、分隔符、带标签的章节，会影响模型可靠使用信息的程度

**一个实践例子：**

一个智能体已经运行了 45 分钟，积累了 80,000 个 token 的对话历史。它的窗口是 128,000。

即使历史逐渐填满窗口，你也不想丢掉原始目标和约束。

上下文工程的做法是：压缩较旧的工具输出，概括早期推理，并在整个会话中让任务定义始终显眼。

提示词工程是在写好指令。

上下文工程是在构建让这些指令真正被遵守的环境。

## **这 6 个概念如何组成一个系统**

![](https://feishu.cn/file/Iyd0bPy0SoCP1GxB3ubcioY5n7c)

*记忆 → RAG + Embedding（系统知道什么）*

*思考 → LLM + token + 上下文窗口（系统如何基于所知进行推理）*

*行动 → 智能体循环 + 工具（系统能在世界中做什么）*

*评估 → Evals（你如何知道它是否有效）*

*胶水 → 上下文工程（决定上述部分之间流动什么）*

简单聊天机器人只是思考。

客服智能体是记忆 + 思考 + 行动。

可靠的生产系统会加入评估。

真正的复杂度在于这些部件连接得有多好。

**任何单次请求的流程：**

用户问题

→ 上下文工程决定要包含什么

→ Embedding 取回相关记忆（RAG）

→ token 决定窗口里能装下多少内容

→ LLM 基于组装好的上下文进行推理

→ 智能体循环决定是否还需要更多信息

→ 评测度量输出是否真的正确

---

## **从哪里开始**

你不需要一次掌握全部六个概念。

→ 从 token 和上下文窗口开始，它们会影响你构建的一切

→ 当你需要语义搜索或记忆时，再加入 Embedding

→ 当你需要让模型基于自己的数据作答时，学习 RAG

→ 当你需要自动化时，学习智能体循环

→ 在任何东西进入生产前，加入评测

→ 当其他概念变得直觉化后，应用上下文工程

这个顺序不是随意的。

每个概念都会让下一个概念变得可学。

---

## **最终结论**

大多数在生产环境中使用AI遇到困难的团队，并非是因为选错了模型或库。

他们的问题是跳过了这六个概念中的某一个。

智能体永远循环，是因为没人考虑停止条件。

RAG 答案错误，是因为没人测量检索。

提示词在长会话里失效，是因为没人理解上下文窗口如何被填满。

这些并不是高级问题。

它们是基础问题，只是披上了技术词汇的外衣。

工具每六个月都会变。

这六个概念解释的是工具如何工作。

学会这些概念，你就再也不会被新工具搞糊涂。

更重要的是，你再也不会看着一个智能体循环一整夜、烧掉 200 美元，然后还在想到底哪里出了问题。
