我去，刚刚发现 Notion 上线了一个巨有意思的小功能。

在 Notion 的文档里可以直接插入 HTML 块了。不是外链跳转，是直接可以在文档里原生嵌入 HTML。这功能已经正式上线了。

不禁感叹，这就是交互创新。期待飞书文档、钉钉文档、腾讯文档等等，能早点上类似的特性啊。我非常需要。

放了个他们官方的宣传视频，大家看，它可以像插入图片、表格等等一样，在一个文档中插入 HTML。

说起来，这事和另外一篇文章有关。

几个月前 Anthropic 公司 Claude Code 团队的工程师 Thariq Shihipar 写了一篇全网爆火的推文，核心观点就一句话：HTML is the new Markdown。

![](https://mmbiz.qpic.cn/mmbiz_png/iagroyM7YaBmq7RKS4RtleEiavzEVW8cnzpicwrYpt0KMbsAZ8y2cxKgoomLNRFgB2GhTDglAXbWDEeFQuZ9Q8IJRcP5xYJNiaaibnMVJKicicjhN8/640?wx_fmt=png&from=appmsg)

这篇文章发出来之后，大家吵成一锅。

很多人觉得，Markdown 本质上已经解决了大部分写作和沟通问题，标题、列表、代码块、链接，足够轻量和结构化。再往上加 HTML，就等于把问题搞复杂了。

但同时，HTML 的优点也非常明显。Markdown 里的内容是静态的，我们只能读。

HTML 可以加交互，比如折叠、切换视图、滑块参数、对比开关。很多复杂概念，一旦变成可交互的东西，理解成本会下降一大截。

吵了半天，没什么结论。

那次，我从 Anthropic 的文章中，得到的最重要的启发是：HTML 居然可以成为一个信息容器。

通常我认为 HTML 就是网页格式，就是用来构建网页的。

但 Anthropic 的同学换了一个视角来看待 HTML，把它当成了通用的信息容器，在里面可以实现复杂的想法排版和可视化。

这次，Notion 这次把这个思路带到了产品当中。作为一个产品经理，真心觉得人家牛 X。这就是对需求的洞察。

上面这个视频是我导入了一个自己做的 HTML。

如果超过了 Notion 预置的 HTML 容器的话，它会有一个展开的按钮。这一步适配我甚至觉得有点多余。

我测试了一下，只要用 Notion 的 AI 生成的 HTML 都会默认适配这个容器。

虽然只是一个很小的功能，但我觉得很妙。

因为过去一年有一个巨大的变化：所有人都可以快速通过 AI 生成一段 HTML 了，不只是程序员。这意味着 HTML 作为信息载体的供给侧，已经彻底打开了。

很多产品其实也看到了这个趋势，但它们的做法是给用户提供一个免费的托管空间，让大家把 HTML 部署上去，通过链接来分享。

这一步有用，但不够。它还是把 HTML 当成了一个外部的东西，需要跳出文档去看。

Notion 这个功能的判断是，既然 HTML 已经是一个信息容器，那它就应该成为文档的一部分，而不是文档的附件。

这其实是一个挺反直觉的选择。

过去行业的默认路径是：复杂表达 → 做成独立应用 → 脱离文档体系。

你想做个可交互的图表？去开一个新工具。你想做个可调参数的演示？去写一个小应用。表达越复杂，就离文档越远。

但 Notion 在做相反的事：复杂表达 → 留在文档内部 → 成为文档的原生组成部分。

往深了想一步，文字其实只是文档中最基础的一种表达形态。表格、图表、嵌入组件、交互块、HTML，都只是不同粒度的结构单元。

这次 HTML 块的功能，本质上是拓宽了文档能承载的表达边界。

这一点我特别感同身受。最近我在公司内部推进一个文档协作方面的新标准：

所有的文档，不要超过 1000 字。特别是现在有了 AI 之后，大家想把文档写长，这太容易了。写的越长，其他人能够有效接收的效率也就越低。

![](https://mmbiz.qpic.cn/mmbiz_png/iagroyM7YaBnzzPwnlWSQia3T2Zs4sagq1pA7X1URaRztDEpZ5fSuu1iaBv4f901jq6mbibX3wXiaR6b5WwRRznTviaY0LXaO2aCgmyJibiafxHx9oA/640?wx_fmt=png&from=appmsg)

现在我能想到的结构化表达的方式只有表格和流程图之类。但如果文档工具都能支持 HTML 块，那信息表达的空间一下子就打开了。

一个交互式的对比组件，可能比写三段文字更容易让人理解方案之间的差异。一个带参数滑块的演示，可能比一整页说明文档更能让人看懂配置逻辑。

这对协作文档来说是一次重大突破。
