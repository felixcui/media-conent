## 摘要

YC设计负责人E Bufar分享了AI对设计工作流的重写，指出设计师正从传统页面制作转向通过语音快速外化意图，将工作变为生成与微调的反馈回路。她以Paxel项目为例，展示了将编码过程数据化以复盘工作模式的方法，并为人和Agent设计了不同的网页入口，强调面向机器的界面需注重内容结构与上下文传递。这表明设计师的新姿态是准备好上下文并清晰表达想法，从而快速实验并创造出可随时丢弃的工具。

E Bufar 是 Y Combinator 的设计负责人，这期对话里她展示了自己最近做的三个项目：用 Paxel 观察人们如何与编码 Agent 协作，用 Sodazine 记录旧金山的城市故事，再用一套新的设计工具为 Startup School 制作视觉资产。

她几乎不再用传统方式从零开始做页面，而是通过说话、整理上下文、快速生成和持续微调，把设计工作变成一条可以反复实验的反馈回路。对设计师、产品经理和工程师来说，这里面最值得关注的并非某个工具的功能，而是设计师开始拥有了一种新的工作姿势：把想法说清楚，把上下文准备好，再为自己造出一组随时可以丢弃的工具。

## 设计师先把键盘放下，工作会怎样

E Bufar 说，自己现在几乎只在 Conductor 和 Paper.design 里完成一个项目的端到端工作。寻找视觉灵感时，她会回到 Pinterest，给项目建立一个 mood book，把喜欢的图片和大致的视觉感觉放在一起。真正改变工作节奏的是输入方式：她发现自己思考的速度快过打字，于是按住功能键，直接把想做的功能用一段连续的想法说出来。Aqua 会把这段语音传给电脑，Agent 再把它落实成页面和交互。 **设计师的第一步不再是熟练操作工具，而是把脑中的意图尽快外化。** 这让设计师可以更早看到结果，也更早发现自己其实没有想清楚的地方。

她描述的工作状态非常具体：按住功能键，给出一段关于功能、页面和感觉的连续表达，几乎不需要碰鼠标。对于仍然习惯在 Figma、编辑器和文档之间来回切换的人，这个变化的意义在于，想法可以先保持完整，再让 Agent 帮忙拆成可执行的步骤。设计师依旧要判断页面是否正确，只是判断会更早发生在成品出现之前。

“我意识到，自己的思考速度比打字快得多。与其打字，我更愿意直接和电脑说话。”

## Paxel 把编码过程变成可观察的数据

Paxel 的出发点，是 YC 想知道人们究竟怎样使用 coding agents。现在每个人都在摸索自己的技巧：有人偏爱某个模型，有人总在深夜提交代码，有人坚持使用 plan mode，也有人会在 Agent 出错时连续重试。可这些行为大多只留在自己的电脑里，同行之间很难互相看见。Paxel 通过终端命令读取 Claude、Codex 和 Cursor 的编码记录，返回一组有趣的个人报告。它甚至借鉴 Spotify Wrapped，把一次次编码 session 变成可以回看的卡片。 **当设计师开始分析自己的工作记录，经验就从模糊的感觉变成了可以讨论的模式。** 未来积累足够多的数据后，Paxel 还会把一个人的习惯与其他 builder 对比。

第一版 Paxel 仍然是 single-player 模式，因为团队还没有收集足够多的编码记录。报告里的卡片会回答一些很私人、也很具体的问题：你最喜欢哪个模型，最常在什么时候提交代码，是否使用 plan mode，或者什么时候对 Agent 最不耐烦。这样的反馈有一点像工作复盘，也像把原本藏在机器里的个人习惯重新放回视野里。

“我们想为你的每次编码会话，做一个类似 Spotify Wrapped 的年度回顾。”

## 同一套网站，要为人和机器各做一份

在 Paxel 的页面上，E Bufar 特意放了 human 和 machine 两个入口。给人看的版本可以有动效、卡片、悬停反馈和视觉节奏；给 Agent 看的版本则是一份更轻、更容易读取的 Markdown 文件，顶部还会提醒 Agent 不要自动运行页面里的命令。这个选择揭示了一个新的设计问题：当页面的使用者包含机器时，视觉层只是其中一层，内容结构、信息密度和安全提示同样需要被设计。 **面向 Agent 的界面，核心工作是给它准确的内容，让它快速获得上下文并继续行动。** 未来一个产品可能同时拥有面向人的网页和面向机器的内容入口，它们服务同一套产品，却遵循不同的阅读逻辑。

Paxel 的 machine 版本顶部还提供复制按钮，用户可以把整页内容直接放进 Claude Code，再继续提问。功能请求表单也被设计成一个 prompt box，用户可以附上截图和录屏，后台 Agent 会据此打开 PR。人仍然保留最后的合并决定，机器则承担了从需求到实现之间的大量重复工作。设计师在这里设计的，是人与 Agent 之间的信息通道。

“Agent 不关心视觉效果，它更关心的是内容本身。”

## 当一切可改，设计师就开始造自己的工具

为了做 Paxel，E Bufar 让 Claude Code 实现了 Paper.design 的图像抖动 shader，然后又给自己造了一个小型调参面板。她可以在里面反复调整颗粒感、边缘、旋转和缩放，直到页面拥有想要的质感。这个面板并不需要成为正式产品，完成任务后就可以丢掉。她把这种方式称为 disposable design：为了探索一个问题，临时造一个只服务自己的工具。类似的思路也出现在功能请求表单里，用户提交的内容会直接作为 prompt 交给 Agent，由 Agent 开一个 PR，再由人决定是否合并。 **设计师获得的自由，来自可以随时为局部问题造出一件临时工具。** 工具的寿命变短了，尝试的次数却变多了。

她特别提到，Paper.design 的 shader 可以直接通过 Claude Code 使用。调参面板后来还被公开出来，让其他人也能在桌面端试试不同的效果。这个例子让她意识到，设计师需要训练一种新的肌肉：发现局部不对时，马上想到“我可以为自己做一个调节它的工具”，而不是把问题留到下一次正式开发。

“所有东西都可以编辑、移动，也都可以改变。”

## 上下文越完整，Agent 越像合作者

做 Sodazine 时，团队把每一次会议都录下来，再把逐字稿放进一个名为 soul.md 的文件。这个文件记录项目的背景、讨论、manifesto 和后续决策，也可以继续拆成 design.md、content.md 等更细的层级。它承担的是项目的长期记忆，之后每一次设计和开发都可以回到同一份上下文里。E Bufar 反复强调，信息越完整，Agent 越能理解项目真正的气质。 **高质量输出的起点，往往是一份愿意花时间整理的上下文文件。** 这份文件也让团队更容易把隐含的判断交给 Agent，让 Agent 在足够多的背景里做出让人意外、却仍然属于项目的提案。

她没有把 soul.md 当成一份漂亮的项目介绍，而是把它当成 exhaustive glossary：会议中的细节、团队的共识、manifesto 里的语气、设计上的偏好，都可以放进去。文件也能建立层级，让不同主题拥有自己的上下文。她说自己还没找到唯一正确的组织方法，当前更重要的事情，是先把这些信息捕捉下来，让 Agent 有机会读懂它们。

“我们能给 Agent 的上下文越完整，结果就会越好。”

## 16 个版本的价值，在于快速排除

有了 soul.md，E Bufar 没有直接要求 Agent 交付最终网站，而是先用 Pinterest 建 mood board，再把图片、文章标题和视觉方向一起交给 Claude，让它一次生成 16 个版本。为了浏览这些版本，她又做了一个只给自己使用的 glossary 页面，可以把喜欢的方案置顶，避免在大量探索中丢失线索。这里的关键不是一次生成出完美页面，而是让真实内容尽快进入不同的视觉结构。看过 16 个版本后，哪些布局值得保留、哪些元素应该组合，判断会具体很多。 **AI 把第一次尝试的成本压低后，设计师就能把时间放在比较、筛选和组合上。** 这是一种更接近工作室的过程：先大量试，再把好的部分留下来。

她把这些页面称为给自己的 disposable design。它们可以使用真实内容，拥有真实的视觉方向，却不承担公开上线的压力。看到喜欢的方案后，她会把它置顶；看到不喜欢的方案，直接跳过。这样做的好处，是设计师可以在早期同时处理多个方向，不必过早把时间投入到一个还没有被验证的答案上。

“因为这是一次快速生成，所以你不会期待它一开始就有极高的完成度。你是在把它当作探索工具。”

## AI 参与越深，人的取舍越重要

Sodazine 里有一个很重要的边界：实体杂志和其中的平面艺术，团队刻意决定不使用 AI。那些作品看上去高度细节化，也能让人感到创作者花了几个月反复打磨。与此同时，围绕杂志建立的网站、地图、文章入口和分享功能，又成为 Agent 很适合参与的部分。这个区分让设计流程更清楚：有些体验需要保留人的手工、时间和选择，有些部分则可以交给 Agent 快速探索。 **AI 让设计更快，却没有替设计师决定什么值得被认真对待。** 越容易生成，越需要有人为项目设定边界，守住风格，也守住那些不能被自动化抹平的细节。

Sodazine 的网站则延续了另一条线索：人们可以在旧金山地图上匿名留下一个地点和一段记忆，再浏览其他人的小故事。地图、文章入口和分享卡片由 Agent 协助快速实现，杂志本身依旧保留创作者慢慢打磨的质感。两种速度放在同一个项目里，反而让团队更清楚每一部分需要什么样的注意力。

“你能看出来，有人花了几个月时间打磨它。”

## 品牌也会进入同一条代码反馈回路

为 Startup School 做视觉资产时，E Bufar 先在 Figma 里手动摆放嘉宾照片，很快发现接下来还会有更多名字和卡片。于是她让 Claude 做了一个自己的模板工具：名单更新后，卡片可以批量生成；文字布局、shader 参数和动效也能在同一个地方调整。为了让社交媒体上的视频首尾无缝，她又做了一个定时录屏工具，保证循环从同一个像素开始和结束。录取通知里的个性化票券也沿用同一套视觉语言。 **当设计资产进入代码，品牌就获得了可复用、可调参、可持续扩展的结构。** 同一套 shader 可以被搬到社交媒体、门票和现场大屏，视觉一致性因此变成了一个可以由 Agent 协助维护的系统。

Startup School 预计有超过 6000 人参加，嘉宾阵容也会不断更新。对设计团队来说，最麻烦的环节往往不是做出第一张卡片，而是保持几十张卡片、不同社交媒体尺寸和现场屏幕之间的统一。模板工具、参数面板和循环录屏工具把这件事变成了一个可重复的流程。设计师仍然决定橙色渐变、图片比例和整体气质，Agent 则帮助把这些判断稳定地复制到更多触点。

“这会彻底改变我们今后进行品牌设计的方式。”

## 写在最后

E Bufar 的方法可以从一个小实验开始：把下一次会议完整记录下来，整理成一份项目上下文；找几张真正喜欢的参考图，让 Agent 基于真实内容生成多个版本；再为最麻烦的局部造一个只服务自己的工具。设计师的价值会越来越多地体现在判断、取舍和想象力上，而这些能力，恰好需要通过一次次具体的尝试被训练出来。下一步可以回看每一次生成，记录自己保留了什么、删掉了什么，以及哪些决定必须由自己完成。这样的记录会慢慢变成个人的设计方法库。
