## 摘要

本文为工程师提供了一套包含15种模式的生产级AI Agent架构选型速查表，旨在解决Agent从演示走向生产时因不断叠加提示词和工具导致的系统混乱问题。文章强调，只有存在不确定性、需动态决策或迭代的任务才适合引入Agent，简单可预测的任务直接调用模型更高效。文中以单Agent模式和多Agent顺序模式为例，详细剖析了它们的适用条件与失效信号，指导团队根据任务的不确定性层级精准匹配架构模式。

## 选择模式之前

并非每个任务都需要 agent。

一个任务值得上 agent，通常有这几种信号：

- 单次模型调用无法稳定产出可靠结果
- 模型必须在运行时决定该用哪个工具或数据源
- 任务需要规划、校验，或迭代式改进
- workflow 里存在真实不确定性，无法提前硬编码

如果输入到输出的路径很可预测，这类任务通常无需 agent。

比如摘要、分类、简单抽取、模板化生成。

这类任务直接调用模型，通常更快、更便宜，也更可靠。

外面再包一层 agent，只会增加延迟和故障点，收益接近于零。

## 模式 1：Single Agent

这是最简单，也最常见的起点。

一个模型。

一个 system prompt。

一组边界清晰的工具。

模型自己决定调用哪个工具，观察结果，然后继续推进，直到掌握足够信息来回答问题。

真实例子：一个客服 agent，负责查询订单状态、检查物流，并在自己无法解决时创建工单。

整个流程只靠 2 到 3 个工具，加上一份清晰职责。

适用场景：任务定义明确，工具集较小，而且单个 agent 能完整持有上下文而不会混乱。

失效信号：你不断往里加工具，system prompt 长到超过一页。这个信号说明你需要换一种模式，而非继续把 prompt 拉长。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALSc3VwicOiaCDqYT99jj4lSNz8DlnbxicyCa9V8cFQBbRfl4j1vlEaWibVa4H3O1AiccibWULUibgcUoQHiau9ptXIlVcuIq6vQCNjk4jM/640?from=appmsg)

Design Pattern

## 模式 2：Multi-Agent Sequential

多个专用 agent 按固定顺序运行。

前一个 agent 的输出，成为下一个 agent 的输入。

真实例子：合同审查流水线。第一个 agent 抽取义务条款，第二个识别风险，第三个为采购团队起草摘要。整个顺序始终固定。

适用场景：workflow 拥有清晰、可重复的阶段，而且每个阶段的产物正好满足下一阶段的输入需求。

失效信号：流程顺序需要根据中途发现的内容动态变化。顺序式流水线默认路径固定；一旦路径会变，就该换成更动态的模式。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/HR8kIomiaALSEoBbJRSxCWPOQCvO7Id93EJZj8Q306F0HaZhsQh0t9IBkibxkrZzloK2MxTxRM5ac9gJBEgVr84anb0krHS2Grbn0oqIJRlV4/640?from=appmsg)

Design Pattern

## 模式 3：Multi-Agent Parallel

多个彼此独立的子任务同时运行，最后再合并成一个整体视图。

### 真实例子

凌晨 2 点的生产事故。

三个 agents 同时排查日志、指标和最近部署记录，大家并行推进，而非串行排队，因为故障期间每一分钟都很宝贵。

适用场景：子任务确实互相独立，而且速度非常重要。

失效信号：任务之间其实有依赖关系。把有依赖的工作硬塞进并行执行，只会制造竞态条件和不完整上下文。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALTiagibWG4e4c7JP1jqicgcuZNNr8IZib2f7Fg7tPwWqWOLyDiavbqwepiaMot2dxKmIicWziagT4GNL5X5pHfeSxn6MsHmx2sIfasqYNI/640?from=appmsg)

Design Pattern

## 模式 4：Loop

重复执行一串步骤，直到满足退出条件。

### 真实例子

一个数据清洗 agent，先分析混乱的 CSV 数据，再提出清洗方案，然后检查是否达到质量标准。

如果没达标，就继续重试，直到达到轮数上限。

适用场景：任务需要多次尝试，而且你能定义一个清晰、可检查的停止条件。

失效信号：退出条件本身并不可靠。少了这个条件，系统就可能无限消耗成本，甚至永远停不下来。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALSnxW4bgtqxfugJvGWNlz4tgyHostibBIicxfthw8SlFxUefibxvXgscOtPsic7XRGQFMUdB9icmEJWuj2DR3Ec5IkhnutBOlPddqAc/640?from=appmsg)

Design Pattern

## 模式 5：Review and Critique

一个 judge agent 负责审查另一个 agent 的输出，提出批评，并给出可执行的具体反馈。

真实例子：一份生成后的报告，会先交给独立的“critic” agent 审阅。

它会标出论证薄弱、证据缺失或表述含混的部分，然后才会流向人工。

适用场景：质量优先级高于速度，而且你希望第二视角内建在系统里，而非事后外挂。

失效信号：critic agent 和生成 agent 拥有相同盲区。若审查者建立在相近假设上，同类错误往往抓不出来。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALRafUIwDDdGbnop7IHQyIia0tIeiaYIvOzlZick334Is3XsteTxohoUzryLlA2nkBWhFZjExtuAofdmb0x4YGGChukSGkAialMV4Sw/640?from=appmsg)

Design Pattern

## 模式 6：Iterative Refinement

这是一个带质量分阈值的反馈循环。

生成器会持续改写，直到分数越过门槛。

### 真实例子

一个营销文案生成器，会根据品牌规范给自己的草稿打分，然后不断重写，直到达到最低质量分。

这里并非一次性通过/失败判断，而是一个有梯度的持续改进过程。

适用场景：输出质量确实存在明显波动，而且“够好”这件事可以被量化成一个明确阈值。

失效信号：评分函数模糊，或者很容易被模型投机利用。如果模型可以抬高分数，却没有真实改进，这个 loop 只会白白烧掉 token。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALQByiaY8ibQXS8jakytAF9gWlW9t8gxFKCibQcIl0xsnIGRxsy80MfyHRLW1J2h1zKBV3KJSfxoBicQIyA1lhbzsT1QeEaKQqOSqbU/640?from=appmsg)

Design Pattern

## 模式 7：Coordinator

一个中心化路由 agent，根据用户请求的真实内容，把任务分发给专门的 agents。

真实例子：支持工单会被路由到计费、技术、账号、物流或欺诈专员。每个专员只掌握狭窄上下文，避免让一个 agent 试图理解所有事情。

适用场景：请求类型确实差异很大，而且每一类都需要不同的上下文、工具，或决策逻辑。

失效信号：路由本身变得含混。如果请求内容无法清楚落进某一类，coordinator 就会演变成新的瓶颈和误路由来源。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALTic0v3neWwSLGWFIVXrsY88uBibkiaQKE2o3iabkhhOa5GOSlyzY4tfB9R2HGwCQKYPKkAyQjvaVFB34QxwnMCGvkO4uRCz1uTByc/640?from=appmsg)

Design Pattern

## 模式 8：Hierarchical Task Decomposition

一个根 agent 先把复杂目标拆成更小的子目标，再委派给专长不同的 worker，最后把结果综合成一个答案。

### 真实例子

“明年我们该优先进入哪 3 个国家？” 这个问题会被拆成竞争分析、监管研究、物流可行性和市场规模测算。

每一块交给不同 specialist，最后再汇总。

适用场景：问题过于宽，单轮推理装不下，但它可以被清楚拆成多个独立的专业领域。

失效信号：子目标之间其实并不独立。若多个工作流需要实时互相喂信息，前置式拆解会丢失这种互动。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALSfBOs06HDFdUq9fSDgbiaHsTbIoUArrCZjUohbiczH0k1ESUFNQEKTABiak4fRlPWicLJkPf9PDzEMh6EA12xHI8BBhfWIicdmELD8/640?from=appmsg)

Design Pattern

## 模式 9：Swarm

多个 specialist agent 参与同一场共享讨论，互相挑战彼此假设，再由一个 facilitator 综合成最终建议。

### 真实例子

公司该不该推出订阅档位？

研究、工程、财务和支持 agents 会围绕各自立场展开多轮讨论，最后由 facilitator 权衡各种取舍。

适用场景：问题本身没有唯一正确答案，你需要的是一个经得起推敲的决策，而这个决策应当来自真实存在的竞争性视角。

失效信号：你需要一个快速、确定性的答案。Swarm 天生偏慢，也偏探索型；若你追求速度，它就用错地方了。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALTEJFONHa873QAdwweq6jDFy7ibyX58NcGbeyf5WibjROiasPm8h8kibyxksadXibf7aibWJhUq4SLHT4zxrKxTo7TRwz5SzTBc0ojX8/640?from=appmsg)

Design Pattern

## 模式 10：ReAct（Reason and Act）

agent 在推理和行动之间交替前进：先决定该查什么，再调工具，观察结果，然后判断证据是否已经足够。

### 真实例子

“队列处理器看起来卡住了。” agent 会先搜索文档、检查服务健康状态、关联各类发现，然后才提出修复建议。

调查路径并未预定义，它会随着中途发现而改变。

适用场景：通往答案的路径确实无法提前规划，必须根据每一步暴露出的信息动态调整。

失效信号：调查过程拖得很长，却始终无法收敛。推理-行动循环一定要设置上限，否则系统可能掉进无穷探索。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/HR8kIomiaALTMgVUzqpHHibENbfFWBWQBqFQcvchnRM377icfmlxz4Ismtuw0exRTWI120rQ38zhnZWqZPZjeAWD0iaOVN3Q8bsoXknibawQNWII/640?from=appmsg)

Design Pattern

## 模式 11：Human-in-the-Loop

agent 负责调查并提出建议，但凡遇到高风险或含混情况，最终拍板依然由人来做。

### 真实例子

退款审批。低风险、规则清楚的个案可以自动化处理；

金额过高、带有欺诈信号，或涉及政策例外的个案，会暂停并交给人工复核，然后才会完成最终处理。

适用场景：决策背后带着真实的财务、法律或声誉风险，而全自动化当前还无法被接受。

### 失效信号

你把它只当成一个 UI 功能。这个模式本质上是架构问题。

你需要持久化 state、审查人分配、超时处理和升级路径，而非一个简单的“暂停”按钮。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALQq6Dco9hpdK9g6G3tWKCHT2cnNzC6CK5NHjsH7EVSiccFjEJEq3Nuic5GXibgaNqd4TCGDzfibmr7FmQk2GINhTISWNy2VV0fWduI/640?from=appmsg)

Design Pattern

## 模式 12：Plan-and-Execute

planner agent 会先产出一份完整、结构化的计划。

这份计划可以被审阅，也可以被修改。

随后 executor 再按步骤执行。

### 真实例子

“把 worker fleet 从 10 台扩到 20 台实例，确认队列已清空，再更新 runbook。” 执行开始前，整份计划就已经可见。

和 ReAct 相比，它的路径是先规划出来的，而非边走边长。

适用场景：你希望计划在任何动作发生前就能被审阅或批准。这一点对有真实后果的运维操作非常重要。

失效信号：环境变化速度快过计划执行速度。若陈旧计划被盲目执行，后果往往比没有计划更糟。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALT5mUALqHbH7hiaXRickVQSbFt10dHyibqmTDulpUshzGnKoZ9377iczQv9GlG3e4d8StaXAR03bpH1HibEPC4mmhp9kz5evrbibKth0/640?from=appmsg)

Design Pattern

## 模式 13：Reflexion

agent 会评估自己的失败，反思问题出在哪里，并把这段 memory 带进下一次尝试。

### 真实例子

一个代码生成 agent 写出脚本后，脚本在运行时报错。agent 会分析真实错误，记录该修什么，再继续重试。

这样每一轮都会更聪明一点，避免反复踩进同一个坑。

适用场景：失败本身包含信息，而且自我修正能真实提升下一次尝试的质量。

失效信号：失败模式随机分布，或者彼此关联很弱。只有当系统能从失败里提炼出稳定模式时，Reflexion 才会带来收益。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALTZCa4s7hs8zNRZpggsmpMibRZ8RI1r2sBjZ7licXxhfP4eACJHmmQQHBhKOU0f83ok4OBeaf51yicCo73UjqtX6ncicCNl9UzGjD0/640?from=appmsg)

Design Pattern

## 模式 14：Custom Logic

这是一种混合架构。

确定性代码负责那些绝对不能出错的规则。

模型则负责判断、起草，以及例外情况处理。

### 真实例子

退款 workflow 里，购买验证和欺诈检查会走硬编码的确定性规则，绝不会委托给模型；

而面向客户的回复草稿与路由建议，则保留 agentic 特性。

适用场景：workflow 拥有真实分支逻辑，而且会牵涉法律或财务后果。你需要清楚划分：哪些地方必须确定，哪些地方可以灵活。

失效信号：团队把边界搞混，让模型接手那些本该写死的决策。资格判断、权限控制和资金流转，单靠模型拍板风险太高。

![](https://mmbiz.qpic.cn/sz_mmbiz_jpg/HR8kIomiaALQrGRSKN0ibHTeOnuIB59ibUaCgqaicDiankCWLLAibp3ILVmZ9ZibRsCpuLz2hONL4jkUpSaO4TunehtVMlQteXyeenict9icIbibCTn1s/640?from=appmsg)

Design Pattern

## 模式 15：Event-Driven Agent

agent 不会等人来提问。

它会订阅事件流，一旦触发条件出现，就立刻行动。

真实例子：一个欺诈检测 agent，会在可疑交易事件触发的瞬间立即响应，而非等支持工单晚些时候才浮现。等到那时，损失往往已经发生。

适用场景：时机比一切都重要，等待人工发起请求就会错过行动窗口。

失效信号：触发条件定义得很差。带噪声的事件流加上含混触发器，最终会把系统变成一个经常误报的警报器，甚至漏掉真正信号。

![](https://mmbiz.qpic.cn/mmbiz_jpg/HR8kIomiaALQbE6pYl9Qll1BIZI91eaFYjFzlkwjONC1p3VHr5WnrTJ43o4Zx3g7CeCiasLKK82Mic4H0urICGKiaibwGK2KkDxkNTdsVHkurnOE/640?from=appmsg)

## 模式选择：匹配不确定性的形状，而非追逐热度

正确的模式，取决于你手头工作的“不确定性长什么样”：

- 工具该怎么选存在不确定性 → Single Agent 或 ReAct
- 请求该往哪里路由存在不确定性 → Coordinator
- 输出质量存在不确定性 → Review & Critique 或 Iterative Refinement
- 执行路径存在不确定性 → Plan-and-Execute 或 ReAct
- 自我修正方式存在不确定性 → Reflexion 或 Loop
- 业务风险存在不确定性 → Human-in-the-Loop 或 Custom Logic
- 问题结构存在不确定性 → Hierarchical Decomposition 或 Swarm
- 来不及等待请求发起 → Event-Driven Agent

如果任务只需要一次可靠工具调用，swarm 并不比 single agent 更高级。

如果计划走到第三步就已经过期，Plan-and-Execute 也谈不上优于 ReAct。

最可靠的生产系统，追求的并非“自动化越多越好”。

更稳妥的做法，是把 autonomy 放进真正能创造价值的位置，同时在其余位置加上约束。

## 生产级 agentic system 的 10 条规则

1. 1从能工作的最小模式起步。

一个工具契约清晰的 single agent，往往强过一个工具契约松散的 multi-agent system。

1. 1把工具描述写成合同。模型理解工具能做什么，完全依赖描述文本，并不掌握你的主观意图。
2. 2为每次请求设置迭代次数、工具调用次数和花费上限。没有预算边界的 agent，迟早会在账单上变成风险项。
3. 3记录完整 action trace。工具调用、参数、输出、最终决策都要留痕。少了这些，事故排查基本只能靠猜。
4. 4不可逆动作要放在确定性检查或人工批准之后。涉及资金流转或生产变更时，模型绝对不能成为唯一闸门。
5. 5用真实失败案例做 eval，别只看 happy path。happy path 上的正确性只够做原型；

边界情况里的正确性，才配叫产品。

1. 1在 system prompt 变得难以阅读之前，先按职责拆分 prompt。

若 prompt 里开始不断堆叠“遇到 Y 时别做 X”之类条件，说明这个 agent 已经同时承担了两份工作。

1. 1把 multi-agent system 当成分布式系统来看。局部失败、超时、重试和可观测性，都是基础设施级要求。
2. 2模型审查替代不了确定性校验。judge 可以提升质量；测试、权限检查和硬规则才负责兜住正确性。

### 01｜优先选择更简单的模式。原因很直接

省下来的复杂度预算，可以投入到更好的工具、更好的 prompt，以及更好的 eval 上。

一页结论

选型时先问清楚：你的不确定性是在工具、路径、质量、风险，还是跨角色协作。模式只是对这类不确定性的结构化回答。

更稳妥的生产系统，通常从更小的模式起步，再用确定性校验、人工闸门和观测能力把 autonomy 收在该收的位置。
