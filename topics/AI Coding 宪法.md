# AI Coding“宪法”该怎么写？高手划出这5个重点，让你告别“伪智能”代码

> 自从团队用上了AI写代码，Review时血压就没降下来过：命名五花八门、风格千奇百怪、测试零零散散... 到底是工具不行，还是我们不会用？

最近和几位技术Leader聊天，发现大家普遍面临一个困境：**AI生成的代码“能用”，但“不好用”**。

一位来自大厂的朋友吐槽：“我们团队用上了最新的AI编程工具，效率确实提升不少。但问题也来了——新人用AI写的代码和老人写的风格完全不同，Review成本不降反增。更头疼的是，一些业务边界和安全规则的代码，AI经常漏掉或理解错。”

这不是个例。根据Stack Overflow 2025年的调查，虽然84%的开发者都在用AI工具，但46%的人明确表示“不信任AI的输出准确性”。原因很简单：**生产级项目对AI的要求，远不止生成代码片段那么简单。**

今天我们就来聊聊，如何通过编写高质量的 **AI Coding Rules（规则）**，将这位“才华横溢但偶尔走神”的AI实习生，培养成懂你项目、懂你团队的真正伙伴。

## 一、Rules不是“说明书”，是“项目宪法”

很多人把Rules当成项目说明书的简化版，这是最大的误解。

**Rules的本质是“项目宪法”**——它定义了团队与AI协作的根本法，是所有人必须遵守的最高准则。好的Rules应该具备三个特性：

1.  **稳定性**：不轻易变动，一旦确定就长期生效
2.  **强制性**：AI必须严格遵守，没有“差不多”的空间  
3.  **全面性**：覆盖开发、测试、安全、协作的全流程

### 理想与现实的差距
**模糊的规则：**
```markdown
- 项目使用Java，请遵循规范
```

**具体的规则（来自阿里妈妈实践）：**
```markdown
- 后端使用Java 17，Spring Boot 3.x，Pandora Boot。
- 对外接口必须使用`ResultDTO`统一返回格式，错误码请参考`ErrorCodeEnum`类。
- 分布式场景下必须加分布式锁，Redis锁超时时间不超过3秒。
```

看出区别了吗？规则越模糊，AI发挥的空间越大——但通常不是往好的方向发挥。

## 二、一份合格的Rules应该包含哪些内容？

经过对多个落地项目的总结，我们提炼出Rules的“五维架构”，让你的规则体系既有高度又接地气：

### 维度一：身份与边界 —— 让AI知道“我是谁”

**定位：** >“你是电商订单系统的Java后端开发专家，采用增量迭代的开发模式”

**边界：** >“你可自主生成单元测试和基础CRUD代码，但涉及数据库表结构变更、跨服务接口设计必须获得人工确认”

**技巧：** 像带新人一样带AI。先明确它的角色，再划定它的“决策权”范围。这样它既不会畏手畏脚，也不会自作主张。

### 维度二：代码与风格 —— 统一团队的“笔迹”

**技术栈：**
```markdown
- 语言：Java 17
- 框架：Spring Boot 3.x + MyBatis-Plus
- 数据库：MySQL 8.0，必须使用UTF8MB4字符集
```

**编码规范：**
```markdown
- 类名：大驼峰（PascalCase）
- 方法/变量名：小驼峰（camelCase）  
- 常量名：全大写+下划线（CONSTANT_CASE）
- 注释要求：所有public方法必须用JavaDoc标注参数、返回值、异常；复杂业务逻辑需用中文注释说明“为什么这么做”
- **禁止**：避免嵌套三元运算符，优先使用switch或if/else
```

**一个真实的技巧：** 在Rules里@一个项目中的模范文件，比如：
> “关于Controller层的标准写法，请参考`com.example.controller.UserController.java`”

AI最擅长模仿，给个好榜样比一万句说教都管用。

### 维度三：工程与流程 —— 固化团队的“肌肉记忆”

**目录结构：**
```markdown
- API层：`/app/api/`下按业务模块划分
- Service层：`/service/`下对应业务领域
- 工具类：`/utils/`下统一管理
```

**开发流程：**
```markdown
1. 先写技术方案（必须包含测试策略）
2. 按方案分段实现（一个会话只处理一个模块）
3. 提交前必须运行：`mvn clean compile test`
4. 代码覆盖率不低于80%（新代码）
```

**关键命令：**
```markdown
- `mvn clean install -DskipTests` # 快速构建
- `./scripts/security-check.sh` # 安全扫描
- `npm run typecheck` # 类型检查（前端项目）
```

**腾讯CodeBuddy团队的实践：** 他们会把历史踩过的坑写成“避坑指南”放进Rules：
> “注意：session模块的save()方法是异步的，必须await，否则会导致数据不一致。2025年11月因此产生过线上事故。”

### 维度四：质量与安全 —— 守住底线

**测试规范：**
```markdown
- 单元测试：使用JUnit 5，Mockito
- 命名：`被测类名+Test`，如`UserServiceTest`
- 覆盖率：新增代码行覆盖不低于85%，分支覆盖不低于70%
- **强制要求**：AI生成的每个方法都必须附带至少一个正例和一个边界用例测试
```

**安全检查清单：**
```markdown
✅ 禁止硬编码敏感信息（密钥、密码）
✅ SQL语句必须使用预编译防止注入
✅ 用户输入必须进行XSS过滤
✅ 文件上传必须校验类型和大小
✅ 日志中禁止记录密码、令牌等敏感信息
```

**性能红线：**
```markdown
❌ 禁止循环内执行数据库查询（N+1问题）
❌ 单次加载数据量不超过1000条
✅ 耗时操作必须异步化并设置超时
```

### 维度五：产出与协作 —— 让Review不再痛苦

**产出标准：**
```markdown
- 代码必须完整，不留TODO占位符
- 复杂算法需附带流程图或伪代码说明
- 每个新文件头部必须添加`@author AI-Assistant`
- 变更必须更新对应的文档和CHANGELOG
```

**协作约定：**
```markdown
- 每次代码生成后，AI必须提供变更摘要
- 涉及多人协作的模块，必须@相关模块负责人确认
- 所有生成的代码默认需要人工Review后方可提交
```

## 三、Rules编写实战技巧

### 技巧一：从小处入手，迭代优化
不要试图一口气写出完美的Rules。可以从最常见、最痛点的问题开始：

**第一版（基础版）：**
只包含代码风格和基础技术栈要求。

**第二版（加强版）：**  
加入团队特有的命名约定和架构约束。

**第三版（成熟版）：**
涵盖安全规范、性能要求和完整的开发流程。

### 技巧二：给约束，也给参考
单纯的“禁止”会让AI束手束脚。更好的做法是：“不要用A（因为有问题），推荐用B（参考文件链接）”。

### 技巧三：保持精简，模块化拆分
如果你的项目特别复杂，不要把所有Rules塞进一个文件。可以按模块拆分：
.rules/
├── index.md # 索引，说明何时加载哪些规则
├── java-style.md # Java编码规范
├── security.md # 安全规范
├── api-convention.md # API设计约定
└── workflow.md # 开发流程


在索引文件中这样写：
> “当编码涉及API调用时，必须阅读`api-convention.md`；涉及用户输入处理时，必须阅读`security.md`”

### 技巧四：Rules + Spec + RAG，三维一体
Rules是**硬约束**（必须遵守），但项目知识远不止于此。成熟的团队会构建三维知识体系：

1.  **Rules（规则库）**：静态的、必须遵守的约束
2.  **Spec（规格文档）**：具体的需求实现方案（如OpenSpec格式）
3.  **RAG（检索增强）**：动态的、可检索的历史方案和业务知识

**腾讯某团队的实践：** 他们将十年积累的业务术语、异常处理模板、组件使用案例构建成RAG知识库。当AI编写支付相关代码时，不仅能读到Rules里的技术规范，还能检索到历史上“支付超时重试≤3次”、“幂等性保障方案”等关键业务约束。

## 四、避开这些常见坑

### ❌ 坑一：追求大而全
有的团队Rules写了上万字，涵盖从架构哲学到命名美学的方方面面。结果AI要么忽略大部分内容，要么在生成代码时顾此失彼。

**✅ 正确做法：** Rules控制在500行以内，聚焦最高频、最重要的约束。

### ❌ 坑二：模糊不清的指导  
“优化一下性能”、“保证安全”——这种提示等于没说。

**✅ 正确做法：** “将响应时间从200ms降低到100ms以内”、“所有用户输入必须经过XSS过滤和SQL注入检测”。

### ❌ 坑三：不更新不维护
Rules一旦写好就束之高阁，随着项目演进逐渐失效。

**✅ 正确做法：** 建立Rules的迭代机制。每次Code Review发现AI的共性错误，就思考：“这个约束能否加到Rules里防止再犯？”每季度进行一次Rules Review。

### ❌ 坑四：人在圈外，全靠AI
有些开发者过于依赖AI，自己对Rules里的规范一知半解。

**✅ 正确做法：** Rules的编写和评审应该是团队共识的过程。每个人都应该理解并认可这些约束，这样才能在Review时准确判断AI是否合规。

## 五、写在最后：Rules的灵魂是人

Rules再完美，也只是工具。真正决定AI编码质量的，依然是背后的团队。

一位资深架构师说得很好：“我们用Rules，不是为了限制AI的创造力，而是为了让团队的创造力形成合力。当每个人都用同一套语言、同一套标准与AI协作时，1+1才能真正大于2。”

在这个AI重构研发流程的时代，写好Rules的最大价值或许在于：它逼着我们停下来思考——到底什么是好代码？什么是好的工程实践？然后把思考的结果，固化下来，传承下去。

**规则不是枷锁，而是高效协作的蓝图。**

试着为你的团队写一份AI Coding Rules吧。不必完美，但要开始。你会发现，当你认真对待规则时，AI才会认真对待你的代码。
